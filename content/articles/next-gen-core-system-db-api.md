## 導入：基幹システムの「画面」に何千万円も払っていませんか

基幹システムのリプレイス案件。見積もりが出てきます。

画面設計、画面開発、画面テスト。見積もりの半分以上が「画面」に関わるコストです。

入力画面、一覧画面、承認画面、帳票出力画面。何十画面、何百画面。それぞれにデザイン、バリデーション、ユーザビリティテスト。

で、その画面、現場で本当に使われていますか？

30年、100社以上のシステム導入を見てきた私の実感を言います。

**導入された画面の半分は、結局エクセルに転記されて使われています。**

現場担当者は、基幹システムに入力した後、同じデータをエクセルにコピーして、自分のフォーマットで管理している。なぜか。基幹システムの画面が、自分の業務フローに合っていないからです。

何千万円もかけて作った画面が、エクセルへの転記元になっている。

この構造自体がおかしい。

## 1. あるべき次世代基幹システムの姿

結論から言います。

**次世代の基幹システムは、データベース＋API群です。画面は不要です。**

正確に言えば、「ベンダーが画面を作る」必要がない。

データベースにデータが正しく格納され、APIで読み書きできる。これが基幹システムの本体です。画面は本体ではありません。画面はクライアント、つまり「データの見せ方・触り方」に過ぎません。

この切り分けがわかると、基幹システムの設計思想が根本から変わります。

**基幹 = データベース＋API＋ビジネスロジック**
**画面 = 各自が自分の業務に合わせて作るもの**

データの整合性、トランザクション管理、権限制御、監査ログ。これらはデータベースとAPI層で担保します。ここにこそ投資すべきです。

一方、入力画面のレイアウト、一覧の並び順、どの項目を最初に見たいか。これは人によって、部署によって、業務によって違います。万人共通の「正解の画面」は存在しない。だから画面をベンダーが作り込むこと自体が、構造的にムダなのです。

## 2. エクセルは死なない。むしろ中心になる

「エクセルを脱却しよう」

DXの文脈で、もう何万回も語られてきたフレーズです。

私は逆のことを言います。

**エクセルは生き残ります。それどころか、次世代基幹システムの主要クライアントになります。**

なぜか。

エクセルは、世界で最も普及した「自分専用画面」だからです。

現場の人間は、エクセルで自分の業務に最適化されたフォーマットを、すでに持っています。何年もかけて改良してきた、自分だけのレイアウト、自分だけの計算式、自分だけの管理表。これを捨てさせる必要がない。

次世代基幹システムでは、こうなります：

- **データベースにマスターデータと業務データが格納されている**
- **APIでデータを読み書きできる**
- **エクセルからAPIを叩いて、データを取得・更新する**
- **各自が自分のエクセルフォーマットで作業する**

基幹システムのデータを、自分のエクセルに引っ張ってきて、自分のフォーマットで見る。編集したらAPIで書き戻す。これでいい。

画面開発コストがゼロです。

しかもAI時代には、エクセル以外のクライアントも各自が作れます。Webアプリ、モバイルアプリ、ダッシュボード。Claude CodeやCursorを使って、自分の業務に最適化された入力画面を、自分で作ればいい。

**基幹システムが提供するのはデータとAPIだけ。クライアントは各自がAIで勝手に作る。**

これが次世代の姿です。

## 3. SIPO分析で見えてくる「本当に必要なもの」

ここでSIPOフレームワークの出番です。

業務分析といえばIPO──Input、Process、Outputの3要素で整理するのが基本です。しかし私は、IPOでは足りないと考えています。Inputの手前にある**Source（情報源）**を定義しないと、業務の全体像が見えないからです。

- **S**（Source）：情報がどこから発生するか
- **I**（Input）：処理に必要な情報は何か
- **P**（Process）：どう処理・判断するか
- **O**（Output）：何を生成するか

このSIPOで基幹システムを分析すると、面白いことがわかります。

### Sourceは現場に散在している

建設業の例で考えます。工事の原価管理。

- **S**: 発注書（PDF）、請求書（紙→スキャン）、作業日報（Excel）、口頭報告（現場）、工程表（社内システム）

Sourceは統一されていません。紙もデジタルもある。人の口頭もある。これが現実です。

### InputとOutputはデータベースで定義できる

- **I**: 工事番号、費目コード、金額、日付、承認状態
- **O**: 原価集計表、予実対比表、月次報告書

InputとOutputは、データベースのテーブル構造として明確に定義できます。これが基幹システムの本体です。

### Processは二層に分かれる

- **ビジネスロジック層**（API側）: 承認ルール、集計ロジック、整合性チェック → 基幹システムが担保
- **操作層**（クライアント側）: データの入力、確認、修正 → 各自がエクセルやAI製アプリで行う

### 画面が不要な理由がSIPOで証明される

SIPOで整理すると明確になります。

基幹システムが責任を持つべきは、**I（入力データの定義）とO（出力データの定義）とP（ビジネスロジック）**です。これらはすべてデータベースとAPIで実現できます。

S（情報源の収集）と、P（操作としてのデータ入力）は、現場の人間がやることです。そしてそのインターフェースは、各自が自分の業務に合わせて選べばいい。エクセルでも、自作アプリでも、AIチャットでも。

**画面を基幹システムの一部として作り込む必要がない**──SIPOで分析すれば、この結論に自然と到達します。

## 4. 業務理解と全体コーディネートの時代

ここまで読んで、こう思った方もいるでしょう。

「画面を各自が作るなら、全体の整合性は誰がとるのか？」

その通りです。ここが最も重要なポイントです。

**データベース設計、API設計、ビジネスロジックの定義。これらの「全体コーディネート」が、今まで以上に重要になります。**

画面開発に費やしていたリソースが不要になる分、以下に集中投資できます：

- **データモデリング**：業務データの正しい構造は何か
- **API設計**：どんな操作を、どの粒度で、誰に許可するか
- **ビジネスルール定義**：承認フロー、計算ロジック、バリデーション
- **Source整理**：散在する情報源をどう収集・統合するか

これらはすべて、**業務を深く理解していないとできない仕事**です。

テーブル設計ひとつとっても、建設業の原価管理と製造業の原価管理では、まったく構造が違います。どの粒度で費目を分けるか、変更管理をどう扱うか、概算と確定をどう持つか。業務を知らなければ設計できません。

つまり、次世代の基幹システム構築で最も価値がある人材は、プログラマーではありません。

**業務を理解し、データ構造を設計し、全体をコーディネートできる人間です。**

社内にそういう人材がいなければ、コンサルタントに頼むのが正解です。画面の見た目を議論する会議に時間を使うより、データモデルとAPI設計に投資した方が、はるかにリターンが大きい。

## 5. 基幹システムの未来図

まとめます。

**従来の基幹システム：**
- ベンダーがDB＋ロジック＋画面をフルセットで納品
- 画面コストが全体の50%以上
- 現場に合わず、結局エクセルに転記される
- 改修のたびに追加費用

**次世代の基幹システム：**
- DB＋API＋ビジネスロジックが本体
- 画面は各自がエクセルやAI製アプリで作る
- データの正しさはAPI層で担保
- クライアントは自由に追加・変更できる

この世界では、エクセルは「脱却すべきレガシー」ではありません。**最も柔軟なクライアント**として、堂々と基幹システムのフロントエンドを担います。

そして、AI時代にはエクセル以外のクライアントも個人が作れる。データベースはNeonやSupabase。デプロイはVercel。認証はClerk。APIでつなげば、自分専用の業務アプリが数日で完成する。

**基幹システムはデータとAPIに徹する。クライアントは現場が作る。全体設計はプロがやる。**

これが、30年のシステム導入経験と、SIPOフレームワークから導き出した、次世代基幹システムのあるべき姿です。
