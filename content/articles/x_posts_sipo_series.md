# X投稿用テキスト：SIPOシリーズ

## スレッド①（Short①公開日に投稿）

### ポスト1/4

建設業でAIエージェントが入らない理由は、AIの性能ではない。

情報源（Source）が定義されていないからだ。

図面、口頭、LINE、メール、写真、Excel。
同じ案件の情報がバラバラに散らばっている。

🧵

### ポスト2/4

だからInputがそろわない。
Inputがそろわないから、ProcessもOutputも曖昧になる。

RPAで壊れた仕事は、AIエージェントでも壊れる。
自動化対象が不安定なままでは、技術を変えても失敗構造は同じ。

### ポスト3/4

建設業に必要なのはAI導入ではなく、情報源の再設計。

IPOではなくSIPOで考える必要がある。

S（Source）→ I（Input）→ P（Process）→ O（Output）

まず情報源を定義し、次にInput・Process・Outputを整理する。

### ポスト4/4

建設業でも全部ダメではない。
見積整理、会議要約、書類チェック、社内検索、報告たたき台。
バックオフィス寄り・定型判断寄りはAIエージェントと相性がいい。

ただし、現場の段取り変更、暗黙知判断、責任を伴う最終決定は人が中心のまま。

## スレッド③（Short③公開日に投稿）

### ポスト1/3

AI時代の業務分析は、Sが8割。

なぜか。
Sourceが明確になれば、Inputが明確になる。
InputとOutputが定義できれば、ProcessはAIがかなり作れるから。

### ポスト2/3

昔は処理手順を細かく人が分解していた。
でもこれからは違う。

どこから情報が発生して、誰が持っていて、何が正なのか。
ここを押さえる方が重要。

業務分析の工数の50%以上はSource分析にかけるべき。

### ポスト3/3

従来の業務分析：I → P → O中心。処理の分解に時間をかける。
AI時代の業務分析：S → I → P → O。Sourceの定義に時間をかける。PはAIが補完しやすい。

Sourceが決まれば、業務は見える。

## バズ狙い単発ポスト

### パターンA
RPAで壊れた仕事は、AIエージェントでも壊れる。

### パターンB
AI時代の業務分析はSが8割。SourceがわかればProcessはAIが作る。

### パターンC
建設業の問題は処理ではない。情報源です。

### パターンD
AIエージェントが使えるかは業界で決まらない。業務の定型性と例外の少なさで決まる。

### パターンE
「AIエージェント万能論」は危険。
建設業では、情報源が散らばり、現場ごとに条件が違い、責任が重い。
いきなり自律実行は無理がある。
必要なのはAI導入の前に、業務と情報源の整理。

### パターンF
建設業で何社もコンサルして気づいたこと。
DXが止まる原因は、ほぼ100%「情報源が曖昧」。
図面、口頭、メール、LINE、写真、Excel。
処理を整理する前に、まずSourceを定義すべき。

## マトリクス図ポスト（画像付き）

### キャプション
AIエージェントが使える業務マトリクス。
建設業でも製造業でも、右下から始めればいい。

### 画像内テキスト（2×2マトリクス）

```
横軸: 業務の定型性（低 → 高）
縦軸: 例外・変動の多さ（多 → 少）

左上（AI難しい）:
  現場采配、職人間調整、天候対応
  異常原因の切り分け、ライン停止判断
  → 人が中心

右上（AI一部可能）:
  品質チェック、安全確認、保全記録
  → 判断補助としてAI活用

左下（AI部分的）:
  見積作成、設計レビュー
  → 情報整理としてAI活用

右下（AIエージェント向き）:
  請求照合、発注処理、会議要約
  書類チェック、社内検索、定型レポート
  → 自動化しやすい
```
