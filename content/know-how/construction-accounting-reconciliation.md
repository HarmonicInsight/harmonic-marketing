ステータス: draft

## 本当に一番のボトルネックはどこか

建設業の会計処理で最大のボトルネックは、**「工事の進捗」と「会計の数字」がリアルタイムに繋がっていないこと**である。

具体的には:

- 現場は「出来高ベース」で仕事をしている
- 経理は「請求書ベース」で仕事をしている
- 経営は「月次決算ベース」で判断したい

この3者の時間軸とデータ粒度がずれているため、**「今、この工事は儲かっているのか？」という最も基本的な問いに即答できない**。

これが一番のボトルネックである。消し込みや伝票処理の問題は、この上流の断絶から生まれる「症状」にすぎない。

### ボトルネックの構造図

```
現場                    経理                    経営
(出来高ベース)          (請求書ベース)          (月次決算ベース)

  出来高70%完了  ──×──  請求書まだ未発行  ──×──  売上未計上
  外注費500万発生 ──×──  請求書来月届く    ──×──  原価未計上
  追加工事発生   ──×──  変更契約未処理    ──×──  予算超過把握できず

        ×＝ 情報が断絶している箇所
```

**結果として起きること:**

1. 月末に「突然」赤字工事が判明する
2. 出来高と請求のタイミング差で資金繰りが読めない
3. 消し込みの時点で「この支払は何の工事？」がわからない
4. 変更契約が経理に伝わるのが遅く、予算管理が後追いになる
5. 工事完了後に初めて利益がわかる（＝管理会計が存在しない）

## 1. 前提: 建設業会計の特殊性

### 一般業種との違い

| 項目 | 一般業種 | 建設業 |
|------|---------|--------|
| 取引期間 | 短期（注文→納品→請求→入金） | 長期（数ヶ月〜数年） |
| 売上認識 | 出荷・納品基準 | 工事進行基準 or 工事完成基準 |
| 原価管理 | 製品原価・仕入原価 | 工事原価（材料・外注・労務・経費） |
| 請求形態 | 都度請求 or 月次請求 | 出来高請求・前払請求・完成請求 |
| 支払形態 | 月末締め翌月末払い等 | 出来高払い・前払・保留金・控除あり |
| 消し込み | 請求額＝入金額が基本 | 一部入金、相殺、控除、保留が常態 |

### 建設業特有の勘定科目

| 勘定科目 | 意味 | 管理の難しさ |
|---------|------|------------|
| 未成工事支出金 | 未完成工事の原価累計 | 工事ごとに正確な原価配賦が必要 |
| 完成工事未収入金 | 完成したが未入金 | 工事完成の判定基準が曖昧になりがち |
| 工事未払金 | 外注・材料の未払い | 出来高認定と支払タイミングがずれる |
| 未成工事受入金 | 前受金・中間金 | 工事の進捗と受入金の対応づけが必要 |

## 2. 2本の業務フロー — 売上側と仕入側

### 売上側フロー（受注→入金消込）

```
受注登録 → 契約締結 → 施工・出来高管理 → 出来高請求 → 入金 → 入金消込
   │          │            │                │          │        │
   ↓          ↓            ↓                ↓          ↓        ↓
 工事番号   契約番号     出来高率/%      請求番号    入金明細   消込済仕訳
 発番       金額確定     原価発生        売上計上    銀行データ  残高更新
```

**各ステップの必須データ**:

| ステップ | 必須データ | Source（原本） |
|---------|-----------|---------------|
| 受注登録 | 工事名、発注者、概算金額、工期 | 見積書、内示書 |
| 契約締結 | 契約金額、支払条件、工期、変更条件 | 契約書 |
| 出来高管理 | 出来高率、実績原価、予定原価 | 出来高報告書、工事日報 |
| 出来高請求 | 請求金額、対象期間、出来高根拠 | 請求書、出来高査定書 |
| 入金 | 入金日、入金額、振込元 | 銀行明細データ |
| 入金消込 | 請求と入金の対応、差額理由 | 消込台帳 |

### 仕入側フロー（発注→支払消込）

```
発注登録 → 注文書発行 → 納品/検収 → 支払請求受領 → 支払予定作成 → 支払実行 → 支払消込
   │          │            │            │              │            │          │
   ↓          ↓            ↓            ↓              ↓            ↓          ↓
 発注番号   注文番号     検収記録     請求番号       支払予定番号   振込データ   消込済仕訳
 工事紐付   金額確定     数量確認     原価計上       資金繰り      実行記録     残高更新
```

## 3. 共通キー設計 — 消し込みDXの最重要テーマ

### キーの階層構造

```
工事番号（マスターキー）
  ├── 契約番号（1工事に対しN契約: 本契約、変更契約1、変更契約2...）
  │     ├── 請求番号（1契約に対しN請求: 出来高1回目、2回目...）
  │     │     └── 入金消込番号（1請求に対しN消込: 全額一致、一部入金...）
  │     └── 入金消込番号（前払充当など、請求を経由しない消込）
  │
  ├── 発注番号（1工事に対しN発注: 外注A、材料B...）
  │     ├── 検収番号（1発注に対しN検収）
  │     │     └── 支払消込番号（1検収に対しN消込）
  │     └── 支払予定番号（1発注に対しN支払予定）
  │
  └── 原価コード（1工事に対しN原価要素: 材料、外注、労務、経費）
```

### 番号体系の設計例

| 番号種別 | 形式例 | 説明 |
|---------|--------|------|
| 工事番号 | `KJ-2026-0001` | 年度＋連番、全システム共通のマスターキー |
| 契約番号 | `KJ-2026-0001-C01` | 工事番号＋契約連番（C01=本契約、C02=変更1） |
| 請求番号 | `KJ-2026-0001-S01` | 工事番号＋請求連番（出来高請求回次） |
| 発注番号 | `KJ-2026-0001-P01` | 工事番号＋発注連番 |
| 支払予定番号 | `KJ-2026-0001-P01-Y01` | 発注番号＋支払予定連番 |

**設計原則**: すべての番号が工事番号を先頭に持ち、番号を見ただけでどの工事に属するかわかること。

## 4. 消し込みルールエンジン設計

### 消込パターン分類

| パターン | 説明 | 自動化可否 | 対応方針 |
|---------|------|-----------|---------|
| **完全一致** | 請求額 = 入金額、相手先一致 | ○ 完全自動 | 自動消込 → 仕訳自動起票 |
| **金額差異あり** | 請求額 ≠ 入金額（振込手数料差引等） | ○ ルール自動 | 許容差額ルール設定（例: 手数料880円以内は自動調整） |
| **一部入金** | 出来高の一部のみ入金 | △ 半自動 | 入金を古い請求から順に充当（FIFO）or 指定充当 |
| **相殺** | 安全協会費、保証金等を差し引いて入金 | △ 半自動 | 相殺科目・相殺パターンを事前登録 |
| **前払充当** | 前受金・前払金を完成時に充当 | △ 半自動 | 前払残高から自動充当候補を提示 |
| **保留** | 瑕疵担保、検査未了等で一部保留 | × 手動 | 保留理由・保留期限を登録、期日管理 |
| **複数工事一括** | 1回の振込で複数工事分を支払 | × 手動 | 明細分解UI、過去パターンからのAI推定 |

### 消込ステータス管理

```
[未消込] → [消込候補あり] → [自動消込済] → [確認済]
                ↓
          [手動消込待ち] → [手動消込済] → [確認済]
                ↓
          [保留中] → [保留解除] → [消込済] → [確認済]
```

## 5. 例外処理の設計パターン

### 5.1 一部支払への対応

```
請求: 工事A 出来高請求 1,000万円
入金: 800万円（出来高査定減額200万円）

→ 消込ルール:
  1. 請求1,000万に対し800万を消込
  2. 差額200万を「出来高差異」として記録
  3. 次回請求時に200万を繰り越し or 調整請求
  4. 差異理由を必須入力（査定減額/保留/その他）
```

### 5.2 相殺への対応

```
請求: 工事B 完成請求 5,000万円
入金: 4,950万円（安全協会費30万＋振込手数料880円を相殺）

→ 消込ルール:
  1. 入金4,950万を売掛金に消込
  2. 安全協会費30万を「法定福利費/安全協力会費」に自動振替
  3. 振込手数料880円を「支払手数料」に自動振替
  4. 合計5,000万880円が請求5,000万＋手数料880円と一致 → 完全消込
```

### 5.3 前払充当への対応

```
前払: 工事C 着工時前払 2,000万円（未成工事受入金）
請求: 工事C 完成請求 8,000万円
入金: 6,000万円（前払2,000万充当済）

→ 消込ルール:
  1. 前払金2,000万を未成工事受入金から売上に振替
  2. 入金6,000万を売掛金に消込
  3. 合計8,000万が請求8,000万と一致 → 完全消込
```

## 6. 月次クロージングチェックリスト

### 締め前チェック（毎月25日まで）

- [ ] 当月出来高報告が全工事で完了しているか
- [ ] 出来高に基づく請求書が発行済みか
- [ ] 受領した請求書が全件入力済みか
- [ ] 前月までの未消込残高に異常がないか

### 消込作業（締日〜翌5営業日）

- [ ] 銀行入金データの取込
- [ ] 自動消込の実行と結果確認
- [ ] 手動消込（自動消込できなかった件の処理）
- [ ] 保留案件の状況確認と必要に応じた処理
- [ ] 未消込残高の確認と原因調査

### 月次決算（翌5〜10営業日）

- [ ] 工事別損益の確認（予算対比）
- [ ] 未成工事支出金の残高確認
- [ ] 完成工事未収入金の回収状況確認
- [ ] 工事未払金の計上漏れ確認
- [ ] 進行基準適用工事の出来高率見直し

## 7. DX導入の正しい順序

```
Step 1: 工事番号体系の統一（1-2ヶ月）
  └── 既存番号の棚卸し → 統一ルール策定 → マスター整備

Step 2: 2本のフロー定義（1ヶ月）
  └── 売上側・仕入側の業務フローを文書化 → 各担当の役割明確化

Step 3: 消込ルール定義（1ヶ月）
  └── 例外パターン洗い出し → ルール化 → 自動/半自動/手動の仕分け

Step 4: システム設定/開発（2-3ヶ月）
  └── マスター設定 → ルールエンジン設定 → テスト消込

Step 5: 並行運用（1-2ヶ月）
  └── 旧運用と並行 → 差異確認 → 運用調整

Step 6: 本番移行（1ヶ月）
  └── 完全移行 → 残課題対応 → 定着化
```

**絶対にやってはいけない順序**:

```
× ワークフローツール導入 → 業務フロー定義
  （ツールに合わせた業務になり本末転倒）

× AI-OCR導入 → 番号体系整理
  （読み取れても紐づけられない）

× 自動消込ツール導入 → 例外ルール定義
  （例外だらけで自動化率が上がらない）
```

## 付録: 用語集

| 用語 | 読み | 意味 |
|------|------|------|
| 消し込み | けしこみ | 請求と入金（または債務と支払）を照合し、残高を消す処理 |
| 出来高 | できだか | 工事の進捗に応じた完成割合・金額 |
| 未成工事支出金 | みせいこうじししゅつきん | 未完成の工事にかかった原価の累計額 |
| 完成工事未収入金 | かんせいこうじみしゅうにゅうきん | 完成した工事の未回収売掛金 |
| 工事未払金 | こうじみばらいきん | 工事に関する買掛金・未払金 |
| 未成工事受入金 | みせいこうじうけいれきん | 前受金・中間金（工事完成前の入金） |
| 工事進行基準 | こうじしんこうきじゅん | 工事の進捗に応じて売上を認識する方法 |
| 工事完成基準 | こうじかんせいきじゅん | 工事完成時に一括で売上を認識する方法 |
