# 建設業・ハウジング業における理想のバックオフィス層 — SIPOフレームワーク

作成日: 2026-03-21
カテゴリ: コンサルティング知見 / バックオフィス改革

---

## 1. バックオフィスは「事務処理部門」ではない

理想のバックオフィスは、請求や経費精算を後追いで処理する部署ではない。
本来は、**現場・営業・設計・調達・経理をつなぐ"情報統制層"** である。

特にハウジング業では、

- 案件ごとの個別性が高い
- 契約変更が多い
- 顧客要望が途中で変わる
- 設計変更が原価・工程・調達に連動する
- 完成後の引渡・保証・アフターまで長い

ため、現場だけでなく**情報の流れそのものを統制する層**が必要になる。

---

## 2. IPOではなくSIPOで見るべき理由

### IPOだけで設計すると起きること

| IPO設計 | 表面上 | 実際の問題 |
|---------|--------|-----------|
| Input | 契約情報を入れる | その情報は営業の口頭か？見積書の最新版か？顧客承認済みか？ |
| Process | 承認する | 何をもって正しいと判断したか不明 |
| Output | 発注書を出す | 根拠が曖昧なまま処理される |

**Inputは定義できても、Inputの正当性を保証するSourceが定義されていない。**
だからSが必要になる。

## 3. SIPOの定義

### S：Source（情報の発生源）

情報が最初に発生した場所・人・契機・原本。

| Source例 | 説明 |
|----------|------|
| 顧客の要望 | 口頭・メール・打合せメモ等 |
| 契約書 | 最終合意の法的根拠 |
| 見積承認 | 金額確定の根拠 |
| 設計図面 | 仕様・数量の根拠 |
| 現場実績 | 進捗・出来高の根拠 |
| 検査結果 | 品質確認の根拠 |
| 協力会社報告 | 外注実績の根拠 |
| 引渡確認 | 売上計上・保証開始の起点 |
| アフター受付 | 不具合対応の起点 |

> **重要**: Sourceは単なる「元データ」ではなく、**どの事実を正とするか**を決めるもの。

### I：Input（取込項目）

Sourceから業務システムや管理帳票に取り込む項目。

- 契約金額、変更金額
- 着工予定日、完了日
- 発注数量
- 検査結果、引渡日
- 不具合内容

> **原則**: Inputは**Sourceの写し**であり、現場や事務が自由に作るものではない。

### P：Process（処理・判断）

判断、承認、変換、照合、集計、進捗化、会計処理。

- 契約承認 → 予算化 → 発注化
- 出来高計上 → 売上計上 → 原価配賦
- 設計変更 → 金額変更 → 工程変更 → 再承認
- 完了 → 検査 → 引渡 → 保証開始
- 変更履歴管理、保証対応判定

### O：Output（成果物）

意思決定や現場実行のために使う成果物。

- 工事台帳、発注書、支払一覧
- 原価予実表、進捗レポート
- 顧客向け案内、引渡資料
- アフター対応一覧
- 経営ダッシュボード

## 4. なぜSourceが抜けると全てが壊れるのか

建設業やハウジング業で起きている問題の多くは、処理能力の不足ではなく、**Source不在のままInputだけが回っていること**にある。

### 典型的なSource曖昧性の例

| 情報 | 候補A | 候補B | 問題 |
|------|-------|-------|------|
| 見積金額 | 営業ファイル | 基幹システム | どちらが正か不明 |
| 契約後変更 | 議事録 | 覚書 | 根拠が分散 |
| 着工判断 | 現場所長の連絡 | 社内承認 | 権限が曖昧 |
| 出来高 | 写真 / 検査記録 / 日報 | — | 原本が不統一 |
| 売上計上 | 引渡書 | 施主検収 | タイミングがずれる |

### 最も深刻な症状：「鉛筆を舐める」問題

現場で最も多いのは、**1人の担当者が複数の帳票を見ながら、自分の頭の中で数字を作っている**状態である。

典型例:
- 5つの帳票（契約書・変更指示書・発注台帳・出来高報告・請求書）を並べる
- それぞれの数字が微妙に合わない
- 担当者が経験と勘で「正しいはずの数字」を判断し、最終帳票に書く
- その判断ロジックは本人の頭の中にしかない

これが**「鉛筆を舐める」**という状態であり、問題の本質は3つある。

| 問題 | 内容 |
|------|------|
| **ロジックのブラックボックス化** | 5つの帳票からどう数字を導いたか、本人以外に再現できない |
| **検証不能** | 上司も経理も、出てきた数字が正しいか確認する手段がない |
| **引継ぎ不能** | 担当者が異動・退職すると、数字の作り方ごと消える |

つまり、Sourceが定義されていないと、**複数の帳票の間を埋める作業が発生し、その埋め方が個人技になる**。
これは「属人化」よりさらに深刻で、**組織として数字の正当性を保証できない状態**である。

> Sourceが1つに定まっていれば、そもそも5つの帳票を突き合わせる必要がない。
> 「鉛筆を舐める」必要があるということは、Source定義が失敗している証拠である。

### なぜ「鉛筆を舐める」担当者を責められないのか

重要なのは、これは担当者の怠慢ではないという点である。

- 会社がSourceを定義していない
- どの帳票が正かを会社として決めていない
- 結果として、担当者が自分で判断するしかない

つまり、**組織の設計不備を個人の努力で埋めている状態**であり、
むしろその担当者がいなくなったときに初めて問題が顕在化する。

### 結果として起きること

- 二重入力
- 確認作業の増加
- 部門間の責任転嫁
- 数字の不一致
- 手戻り
- 締め処理の遅延
- **特定の担当者に業務が集中し、その人がボトルネックになる**
- **数字の根拠を問われたときに説明できない（監査リスク）**

> **情報源が曖昧な会社ほど、人が頑張って埋めるしかなくなり、属人化する。**
> **そして最も頑張っている人が、最も替えの利かないリスクになる。**

### 解法：AI による帳票間ロジックの逆算

「鉛筆を舐める」ロジックは、担当者にヒアリングしても出てこない。
本人も無意識にやっていることが多く、「経験でわかる」「見ればわかる」としか説明できない。

だからアプローチを変える必要がある。
**システムから出てくるOutputと、現場で実際に使われている帳票を突き合わせて、AIにロジックを逆算させる。**

#### なぜこの方法しかないのか

| 従来のアプローチ | 限界 |
|-----------------|------|
| 担当者にヒアリング | 本人が言語化できない。「いつもこうやっている」で終わる |
| 業務マニュアルを読む | マニュアルと実態が乖離している。そもそもマニュアルがない場合も多い |
| 業務フロー図を書く | 処理の流れは書けるが、**数字の導出ロジック**までは表現できない |
| システムログを見る | システムに入る前の判断（どの数字を採用したか）は記録されていない |

つまり、**ロジックの痕跡は「帳票の数字の対応関係」の中にしか残っていない。**

#### AI分析で何をやるか

具体的には、以下のステップになる。

**Step 1：帳票の収集**
- システムから出力される帳票（公式Output）
- 担当者が実際に使っているExcel・手書き帳票（現場帳票）
- 両方を案件単位で複数セット集める

**Step 2：項目の突合**
- 公式Outputの項目と、現場帳票の項目を並べる
- AIに「この2つの帳票の間で、どの項目がどの項目に対応しているか」を分析させる
- 一致する項目、不一致の項目、現場帳票にしかない項目を洗い出す

**Step 3：ロジックの推定**
- 現場帳票の数字が、公式Outputのどの数字からどう導かれているかをAIに推定させる
- 例：「現場帳票の"調整後原価"は、システムの"発注額"＋"変更指示額"×0.9で近似できる」
- 複数案件のデータで検証し、パターンを特定する

**Step 4：隠れSourceの発見**
- AIの分析で「この数字の根拠がどの帳票にも見当たらない」という項目が出てくる
- それが**担当者の頭の中にしかないSource** — つまり定義すべきだったのに定義されていなかった情報源
- ここを明文化することが、Source定義の核心になる

#### この分析が経営的に意味を持つ理由

| 効果 | 内容 |
|------|------|
| **ブラックボックスの解消** | 数字の導出ロジックが可視化され、担当者以外でも検証可能になる |
| **引継ぎ可能化** | ロジックが文書化されるため、異動・退職に耐えられる |
| **監査対応** | 数字の根拠を第三者に説明できるようになる |
| **Source定義の材料** | 「本当に必要なSource」が逆算的に特定される |
| **システム改善の要件** | 現場帳票にしかない項目＝システムに足りない機能が明確になる |

> 重要なのは、AI分析の目的は「自動化」ではなく「解明」である。
> まず隠れたロジックを明らかにし、そのうえでSourceを再定義する。
> 順序を間違えてシステム化を先にやると、また同じ属人化が起きる。

## 4.5. AI時代のバックオフィス基盤：TDWH（テキストデータウェアハウス）

> **注**: TDWH の技術仕様は `cross-lib-insight-common` に定義済み。
> - 設計標準: `standards/TDWH.md`
> - 型定義: `config/tdwh/types.ts`
> - IPO変換ブリッジ: `config/tdwh/ipo-bridge.ts`
> - AIエージェント: `config/ai-agent-runtime.ts` / `config/orchestrator.ts`
>
> 本章ではSIPOフレームワークの文脈から、なぜTDWHが必要かを論じる。

### なぜデータベース中心だと失敗するのか

従来のバックオフィスシステムは、**リレーショナルデータベース（RDB）を中心**に設計されてきた。

```
従来の構造：
  契約書 → [人が項目を抜き出して入力] → DB → [クエリ] → 帳票
  議事録 → [人が要点を転記] → DB → [クエリ] → 帳票
  メール → [人が判断して入力] → DB → [クエリ] → 帳票
```

この構造の根本的な問題は3つある。

| 問題 | 内容 |
|------|------|
| **情報の欠落** | DBのカラムに収まらない情報（文脈、経緯、ニュアンス）が捨てられる |
| **入力コスト** | 原本からDBへの転記に人手がかかり、そこで歪みが入る |
| **Sourceとの断絶** | DBに入った時点で原本との紐づけが切れ、「なぜこの数字か」が追えなくなる |

つまり、**DBに構造化して入れること自体が、Sourceを殺す行為**になっている。
これが「鉛筆を舐める」問題の構造的原因でもある。

### TDWH：テキストをテキストのまま扱う4層アーキテクチャ

AI時代の正解は、**データベースではなく、TDWH（テキストデータウェアハウス）を中心に置く**ことである。

TDWH は `insight-common` で定義された4層構造を持つ。

```
┌───────────────────────────────────────────────────────────────┐
│  Layer 1: データレイク (Data Lake)                              │
│  契約書・議事録・メール・指示書・報告書を一切加工せず保存          │
│  → RawDocument（OCR文字化け・フィラーワードもそのまま）           │
│                             ↓                                  │
│  Layer 2: キュレーション (Curated)                              │
│  Cleaner + EntityExtractor + Normalizer                        │
│  1生データ → N情報単位に分解、品質スコア付き                      │
│  → CuratedRecord                                               │
│                             ↓                                  │
│  Layer 3: マート (Mart)                                        │
│  目的別に構造化されたナレッジストア                               │
│  Classifier → Chunker → Embedder → VectorDB                   │
│                             ↓                                  │
│  Layer 4: ディスパッチ (Dispatch)                               │
│  IntentClassifier → Router → Integrator                        │
│  → 出典付き統合回答（AIエージェントへ）                          │
└───────────────────────────────────────────────────────────────┘
```

#### SIPOフレームワークとの対応

| SIPO | TDWHでの実現 |
|------|-------------|
| **Source** | Layer 1（データレイク）に**原本そのまま**格納。`RawDocument` として保存され、原本との紐づけが切れない |
| **Input** | Layer 2（キュレーション）で情報単位に分解。`CuratedRecord` が `sourceUrl` と `rawDocumentHash` で原本と紐づく |
| **Process** | Layer 3（マート）＋ IPOブリッジで業務プロセス構造を自動抽出。`ExtractedIpoStructure` として構造化 |
| **Output** | Layer 4（ディスパッチ）が出典付きで統合回答を生成。`IntegrationResult` に `sources` と `gaps` を含む |

#### なぜこれがSourceの問題を解決するのか

| 従来（DB中心） | TDWH中心 |
|---------------|---------|
| 人が項目を抜き出してDBに入力 | テキストをそのまま投入（Layer 1） |
| DB設計が先、業務が後 | 業務の原本が先、構造化は後（Layer 2-3） |
| カラムにない情報は消える | 全文が残る。文脈も経緯も保持される |
| 検索はSQL | 検索はAIによる意図分類＋ベクトル検索（Layer 4） |
| 変更にはDB改修が必要 | テキストは変わらない。キュレーション戦略を変えて再処理可能 |
| Sourceが入力時に断絶 | **Sourceそのものがデータレイクに格納される** |

> 最大のポイント：TDWHでは**生データを一切加工せずLayer 1に保存**する。
> OCR技術が改善されれば再処理可能。キュレーション戦略を変更すれば全データを再構築可能。
> つまりSourceは永続的に保護される。

### 建設業・ハウジング業向けマート設計

`insight-common` のチャンク戦略パターンを建設業に適用すると、以下のマート構成になる。

| マートID | チャンク戦略 | 対応するSource | 用途 |
|----------|-------------|---------------|------|
| `contract` | `case_study` | 契約書・変更契約 | 案件別の契約条件・金額の検索 |
| `design` | `construction_method` | 設計図・仕様書 | 仕様・工法・適用条件の検索 |
| `meeting` | `section` | 議事録・打合せメモ | 決定事項・要望変更の追跡 |
| `cost` | `case_study` | 見積・発注・出来高 | 原価情報の案件横断分析 |
| `inspection` | `safety_topic` | 検査結果・不具合報告 | 品質・安全情報の検索 |
| `regulation` | `legal_article` | 建設業法・契約約款 | 法令・規則の条文検索 |

これにより、AIエージェントが「B邸の変更契約で合意された追加金額はいくらか」と聞かれたとき、
**契約マートと議事録マートを横断検索し、出典付きで回答**できる。

### AIエージェントが個人のタスクを管理する

TDWHを基盤として、`insight-common` で定義されたAIエージェント（`AgentProfile`）と
オーケストレーター（`Orchestrator`）が連動し、バックオフィスの働き方を根本的に変える。

**従来：**
- 朝出社して、自分のタスクを自分で把握する
- 複数のシステムを開いて状況を確認する
- 帳票を突き合わせて数字を作る
- 問題を自分で発見する

**TDWH＋AIエージェント：**
- AIエージェントが全トランザクション・全テキストデータを横断的に把握している
- 朝、AIが「今日あなたがやるべきこと」を提示する
- 問題点（数字の不整合、期限超過、承認漏れ）をAIが先に検知して警告する
- 担当者は判断と実行に集中できる

#### エージェントの動作モデル（`insight-common` 定義準拠）

| `AgentMode` | バックオフィスでの適用 |
|-------------|---------------------|
| `autonomous` | 定型的な照合・集計・アラート生成を自動実行 |
| `human_loop` | 金額確定・承認が必要な処理は人間に確認を求める |
| `watch` | 共有フォルダの契約書・報告書を監視し、自動でTDWHに取込 |
| `schedule` | 日次・月次の締め処理前に自動で不整合チェックを実行 |

#### 具体的にAIエージェントが提示するもの

| 提示内容 | 例 | 根拠 |
|----------|-----|------|
| 今日のタスク | 「A邸の変更契約書が未承認です。3日経過しています」 | 契約マート＋承認ワークフロー |
| 数字の不整合 | 「B邸の発注額合計が予算を12%超過しています」 | 原価マート＋IPO自動抽出結果 |
| 期限アラート | 「C邸の引渡予定日まで2週間ですが、検査日程が未確定です」 | 議事録マート＋工程情報 |
| 関連情報の提示 | 「D邸で同様の仕様変更がありました。そのときの対応メモはこちらです」 | ディスパッチャーの横断検索 |
| 承認待ち | 「あなたの承認を待っている案件が3件あります」 | オーケストレーターの `ApprovalGate` |

### なぜこれが「鉛筆を舐める」問題を解消するのか

```
従来：
  帳票A ─┐
  帳票B ──┼→ [担当者の頭] → 最終数字（ロジック不明）
  帳票C ─┘

TDWH＋AIエージェント：
  テキスト原本群 → [TDWH 4層] → [AIエージェント] → 導出ロジック＋出典ごと提示
                                                   → 担当者は確認・承認のみ
```

AIエージェントは数字を導出する際に、`IntegrationResult` の `sources` として
**どのテキスト（Source）から、どういうロジックで導いたか**を同時に提示できる。
不足情報は `gaps` として明示される。

つまり「鉛筆を舐める」必要がなくなるだけでなく、**ロジックの透明性が構造的に保証される**。

### TDWH → IPO ブリッジ：テキストから業務構造を自動抽出

`insight-common` の `ipo-bridge.ts` は、TDWHに蓄積されたテキストから
IPO構造を自動抽出する仕組みを定義している。

```
TDWH CuratedRecord                    IPO JSON
┌──────────────────┐                 ┌──────────────────┐
│ 経理規程 v2.1    │                 │ IpoData          │
│ 月次決算マニュアル│  → AI 抽出 →   │  └ 経理部         │
│ 定例会議事録     │                 │    └ 月次決算     │
│ ...              │                 │      ├ 原価収集   │
└──────────────────┘                 │      ├ 仕訳入力   │
                                     │      └ 決算承認   │
                                     └──────────────────┘
```

抽出時には `UnresolvedItem` として以下が検出される:
- `unknown_source` — 入力の出どころが不明（＝Source未定義）
- `unknown_destination` — 出力先が不明
- `ambiguous_owner` — 責任者が曖昧
- `missing_kpi` — KPIが未設定
- `unclear_process` — プロセスの内容が不明確

> これはまさに「鉛筆を舐める」問題の自動検出である。
> `unknown_source` が多い業務＝Source定義が失敗している業務であり、
> コンサルタントが最優先で取り組むべきポイントが自動的に特定される。

### データベースを捨てるという意味ではない

誤解のないように補足すると、RDBが不要になるわけではない。

- **会計処理、法定帳簿、マスタ管理** → 引き続きRDBが適切
- **Source管理、業務判断の根拠、経緯の記録** → TDWHが適切

つまり、**中心をどこに置くか**の話である。

従来は「DBが中心で、テキストは添付ファイル扱い」だった。
AI時代は「TDWHが中心で、DBは構造化が必要な処理のための道具」になる。

> 建設業・ハウジング業のバックオフィスは、情報の大半がテキスト（契約書・議事録・メール・報告書）で発生する。
> にもかかわらず、DBのカラムに押し込もうとしてきたことが、あらゆる問題の根源である。
> テキストをテキストのまま扱い、AIに構造化を任せる。これがAI時代のSource管理の正解であり、
> HARMONIC insightはこれを `cross-lib-insight-common` のTDWH標準として既に実装基盤を持っている。

## 5. 理想のバックオフィス4層モデル

### 第1層：Source管理層

情報の発生源を定義し、原本を決める層。

決めるべきこと:
- 何が原本か
- 誰が発生責任を持つか
- いつ発生するか
- どの条件で確定するか
- 変更時は何をもって更新とするか

> ここがない会社は、全部が後工程で崩れる。

### 第2層：Input統制層

Sourceを最小限の入力で業務に取り込む層。

原則:
- 同じ情報を何度も入力しない
- 入力者を増やしすぎない
- 自由記述を減らす
- マスタと紐づける
- 変更履歴を残す

> Inputは"集める作業"ではなく、**Sourceを正しく受け渡す仕組み**であるべき。

### 第3層：Process標準化層

承認・照合・発注・計上・支払・引渡を、案件ごとの勘ではなく標準ルールで処理する層。

連動ルールの例:
- 契約 → 予算 → 発注 → 実績 → 請求 → 回収
- 設計変更 → 金額変更 → 工程変更 → 再承認
- 完了 → 検査 → 引渡 → 売上 → 保証開始

> バックオフィスの理想は、**処理の速さではなく、処理の再現性**。

### 第4層：Output経営活用層

帳票を保管するだけでなく、経営判断に使える形で出す層。

経営者が本来見たいもの:
- どの案件で利益が崩れているか
- 変更契約の未回収はどこか
- 発注漏れや原価超過はどこか
- 引渡遅延の兆候はどこか
- アフター不具合が多い協力会社はどこか

> Outputは帳票ではなく、**意思決定の武器**でなければいけない。

## 6. ハウジング業で特に重要なSource定義

ハウジング業は一般建設よりも顧客接点と変更が多いため、Source定義がさらに重要。

### 6-1. 顧客要望

口頭・メール・打合せメモ・営業メモが混在しやすい。
→ **「正式要望」と「相談レベル」を分ける**ことが必要。

### 6-2. 契約・変更契約

最終合意の根拠を明確にしないと、営業・設計・工事・経理で数字がずれる。

### 6-3. 設計図・仕様書

最新版管理が崩れると、現場も発注も請求も崩れる。

### 6-4. 発注根拠

誰が、どの図面・どの数量・どの予算で発注したか。
→ ここが弱いと原価管理が機能しない。

### 6-5. 現場実績

進捗、出来高、手戻り、不具合、検査結果。
→ 現場個人のメモで終わると**組織学習が起きない**。

### 6-6. 引渡・アフター受付

売上計上や保証開始、不具合責任分界の起点になる。

## 7. コンサルタントの役割

コンサルタントの役割は業務フローを書くだけではない。

本質的には:
- **情報の出生地を見つけること**
- **その情報がどこで歪むかを突き止めること**
- **SourceからOutputまで責任の鎖を設計すること**

### 具体的なアクション

| ステップ | 内容 |
|----------|------|
| 1. Sourceの棚卸し | 何の情報が、どこで、誰によって、どんな形で発生するかを洗い出す |
| 2. 原本の決定 | 同じ情報が複数あるなら、どれを正とするか決める |
| 3. Input最小化 | 現場や事務の転記を減らし、重複入力を止める |
| 4. Processの標準化 | 案件ごとの差ではなく、会社の標準ルールに落とし込む |
| 5. Outputの経営接続 | 帳票を作るための業務から、経営に効く数字を出す業務へ変える |

---

## 8. 経営者向けメッセージ

> 建設業のバックオフィス改革で重要なのは、システム導入ではない。
> 情報の発生源を定義し、現場・営業・設計・経理の間で"何を正とするか"を統一することだ。

> 業務が回らない原因は、担当者の頑張り不足ではない。
> Sourceが定義されていないまま、InputとProcessだけを増やしてきたことにある。

> 理想のバックオフィスとは、事後処理部門ではない。
> 現場を支え、経営判断を支える**情報生産ライン**である。

## 10. HARMONIC insight 技術基盤との接続

このSIPOフレームワークは、`cross-lib-insight-common` で定義済みの技術基盤と直接接続する。

### TDWH（テキストデータウェアハウス）との対応

| SIPOの課題 | insight-common の解決手段 | 定義ファイル |
|-----------|-------------------------|-------------|
| Source定義の欠如 | TDWH Layer 1（データレイク）に原本を保存。`SourceDefinition` で発生源を型定義 | `config/tdwh/types.ts` |
| Inputの重複・転記ロス | TDWH Layer 2（キュレーション）で自動分解。`CuratedRecord` が原本と紐づく | `config/tdwh/types.ts` |
| Processのブラックボックス | IPOブリッジで業務構造を自動抽出。`UnresolvedItem` でSource未定義を自動検出 | `config/tdwh/ipo-bridge.ts` |
| Outputの経営活用不足 | TDWH Layer 4（ディスパッチ）で出典付き統合回答。多次元分析 | `config/tdwh/dimensions.ts` |

### AIエージェント基盤との対応

| SIPOの理想 | insight-common の実現手段 | 定義ファイル |
|-----------|-------------------------|-------------|
| 個人タスクのAI提示 | `AgentProfile` + `AgentMode`（autonomous/human_loop/watch/schedule） | `config/ai-agent-runtime.ts` |
| ジョブの自動実行・承認 | `Orchestrator` + `ApprovalGate` + `WorkflowDefinition` | `config/orchestrator.ts` |
| 帳票ロジックの解明 | `document_analysis` / `data_extraction` スキル | `config/ai-assistant-skills.ts` |

### IPO分析との対応

| SIPOの要素 | insight-common のIPO型 | 定義ファイル |
|-----------|----------------------|-------------|
| 業務ノードの定義 | `IpoNode`（organization/department/process/task） | `config/ipo/types.ts` |
| 実行者の分類 | `ExecutorType`（human/system/ai/hybrid） | `config/ipo/types.ts` |
| DXアセスメント | `DxAssessment` + `AutomationCandidate` + `ProcessIssue` | `config/ipo/analysis.ts` |

### プロダクトとの接続

- **INCA（InsightNoCodeAnalyzer）**: TDWH＋IPOブリッジを使い、テキストから業務構造を自動抽出。`unknown_source` の検出＝Source定義コンサルティングの起点
- **IVIN（InterviewInsight）**: インタビューマート（`interview-mart.ts`）で現場ヒアリングを構造化。`InterviewExtractedProblem` が「鉛筆を舐める」問題を自動分類
- **IAOF（AI業務フロー分析）**: IPO構造の可視化＋DXアセスメント。`humanDependencyRate` でSource不在による属人化度を定量化
- **InsightBot**: AIエージェントランタイム上で動作し、TDWHのディスパッチャーを通じて各担当者にタスク・アラートを提示

### コンサルティングパッケージとしての提供

建設業・ハウジング業向けに、SIPOフレームワークを軸にした**情報統制診断サービス**を提供可能。

```
Phase 1: 現状把握
  IVIN でヒアリング → InterviewMart に蓄積 → 問題の自動分類

Phase 2: Source解明
  帳票をTDWH Layer 1 に投入 → AIで帳票間ロジック逆算
  → IPOブリッジで unknown_source を自動検出

Phase 3: SIPO再設計
  Source定義 → Input最小化 → Process標準化 → Output経営接続
  → IpoData として構造化、DxAssessment で効果測定

Phase 4: AIエージェント導入
  Orchestrator でジョブ定義 → AgentProfile で各担当者にエージェント配置
  → watch/schedule モードで継続的な情報統制を実現
```
