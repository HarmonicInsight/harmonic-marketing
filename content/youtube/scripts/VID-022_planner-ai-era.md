提案書（紙）だけで合意形成しようとするから時間がかかる。
AIを使えば、企画屋が自分でプロトタイプを作って「動くモノ」で提案できる。
これで合意形成のスピードが劇的に変わる。

### スライド3：なぜ提案書だけでは伝わらないのか

**タイトル**: 提案書が伝わらない3つの理由
**箇条書き**:
1. **抽象的すぎる** — 「業務フローを最適化します」では何が変わるか見えない
2. **想像力に依存する** — 読む人がイメージできないと、議論が空中戦になる
3. **フィードバックが曖昧** — 「なんか違う」しか返ってこない

**ナレーション**:
「なぜ提案書だけでは伝わらないのか。3つ理由があります。

1つ目、抽象的すぎる。『業務フローを最適化します』と書いても、具体的に何がどう変わるのか見えません。

2つ目、想像力に依存する。提案書を読む側が、完成形をイメージできるかどうかに依存してしまう。イメージできない人には、いくら説明しても伝わりません。

3つ目、フィードバックが曖昧になる。紙だけだと『なんか違う』『もうちょっとこう……』としか言ってもらえない。何がどう違うのか、具体的な議論ができないんです。」

---

### スライド4：解決策 — 動くモノで提案する

**タイトル**: 解決策: プロトタイプ駆動型の提案
**図解**:
```
従来:  提案書 → 説明会 → 修正 → 説明会 → 修正 → 合意（3ヶ月）

改善:  プロトタイプ → 触ってもらう → 具体的FB → 修正 → 合意（2週間）
```

**ナレーション**:
「解決策はシンプルです。紙の提案書ではなく、動くプロトタイプで提案する。

パワポで『こういう画面になります』と説明する代わりに、実際にその画面を作って触ってもらう。すると何が起きるか。

まず、お客様の理解が一瞬で深まります。『あ、こういうことね』とすぐわかる。
次に、フィードバックが具体的になります。『この項目をここに移動して』『この条件で絞り込めるようにして』と、紙では出てこなかった具体的な要望が出てきます。
結果、合意形成が圧倒的に速くなる。3ヶ月かかっていたものが2週間で済む。」

### スライド5：「でも、プロトタイプを作るのに開発者が要るのでは？」

**タイトル**: ここが変わった — AIで企画屋が自分で作れる
**ステップ図（画面に常時表示）**:
```
① 業務を理解する → ② 最適な流れを提案する → ③ プロトタイプを作る → ④ 見せて改善する
    ↑                                                                    ↓
    └──────────────── フィードバックループ ─────────────────────────────────┘
```

**ナレーション**:
「ここで皆さん思うはずです。『プロトタイプを作るのにエンジニアが要るでしょ？ 結局コストと時間がかかるじゃないか』と。

1年前までは、その通りでした。でも今は違います。AIを使えば、企画屋が自分でプロトタイプを作れます。

私がやっているプロセスを『プロトタイプ駆動型コンサルティング』と呼んでいますが、4つのステップです。

**ステップ1: 業務を理解する。**
お客様の業務がどう動いているか、情報がどこで発生してどこで歪むか、まず自分で理解する。これは企画職やPMなら普段からやっていることです。

**ステップ2: 最適なワークフローを提案する。**
『こういう流れに変えたらどうですか？』と改善案を出す。ここまでは従来と同じ。

**ステップ3: プロトタイプを作る。**
ここが革命です。従来はエンジニアに発注していた部分を、AIに指示して自分で作る。Claude Codeに要件を伝えると、動くプロトタイプが数時間〜数日で出来上がります。

**ステップ4: 見せて改善する。**
動くモノを見せると、お客様から具体的なフィードバックが返ってくる。それをまたAIで反映する。このループを1人で、数日で回せます。」

**タイトル**: 【実演】Claude Codeでプロトタイプを作る
**画面収録**:

**ナレーション**:
「実際にお見せします。

今回は、建設業のバックオフィスで実際にある課題を例にします。『現場の実績データが本社に届くまでに2週間かかる。その間、経営判断ができない』という問題です。

（操作しながら）Claude Codeに、こう伝えます。
『現場の日報データを入力すると、リアルタイムで本社の管理画面に反映されるダッシュボードを作ってください。入力項目は、工事名、作業内容、人工数、進捗率です。』

ポイントは、私はコードの書き方を指示していません。『何を作りたいか』『何が入力で、何が出力か』を伝えているだけです。これが要件定義の力です。

（結果を見せながら）はい、こうなりました。もちろんこのまま本番で使うわけではありません。でも、お客様にこれを見せたらどうでしょう？ 『あ、こういう感じね。でもここは工種ごとに分けたい』と、具体的なフィードバックが出てきますよね。これを紙だけで引き出すのは不可能です。」

### スライド7：なぜ企画屋にこそ向いているのか

**タイトル**: なぜ「企画屋」にこそ向いているのか
**比較表**:

| 能力 | 企画屋が持っている | AIが補ってくれる |
|---|---|---|
| 何を作るべきか判断する | ✅ 業務理解・課題発見 | - |
| どう作るか設計する | ✅ 要件定義・設計力 | - |
| コードを書く | ❌ | ✅ AIが書く |
| 品質を判断する | ✅ 保守運用の経験 | - |
| お客様にフィードバックを求める | ✅ コミュニケーション力 | - |

**ナレーション**:
「なぜこの方法が企画屋やPMに向いているのか。表で整理しました。

何を作るべきか判断する力、設計する力、品質を判断する力、お客様からフィードバックを引き出す力。これは全部、企画職やPMが普段からやっていることです。

唯一できなかったのが『コードを書く』こと。でも、そこだけAIが補ってくれるようになった。

つまり、5つのうち4つは既に持っていて、残り1つをAIが埋めてくれる。だから企画屋にこそ向いているんです。

逆に、コードだけ書ける人は、他の4つが足りない。だからAIがコードを書いてくれても、『何を作るべきか』がわからない。結果として、使い物にならないプロトタイプが出来上がってしまいます。」

『エンジニアの仕事を奪うのでは？』むしろ逆です。プロトタイプで要件が明確になるので、本開発に入った時の手戻りが激減します。エンジニアにとってもありがたい話です。

『自分にもできる？』要件定義をやったことがある方なら、すぐに始められます。コードを書く必要はありません。AIに『何を作りたいか』を伝える力、これが要件定義そのものです。」
