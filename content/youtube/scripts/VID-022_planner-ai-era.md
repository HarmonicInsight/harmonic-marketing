## この動画で伝えたいこと

28年間「企画して、設計して、人に作ってもらう」側だった人間が、AIの登場で「自分で全部作れる」ようになった。
これは単なるツールの話ではなく、キャリアの構造変化。上流経験者こそAI時代の最適解を出せる。

### スライド4：なぜ上流経験者が最強なのか

**タイトル**: 上流経験者が AI時代に最強な5つの理由
**箇条書き**:
1. **何を作るべきかわかる**（企画力）→ STRAVIS 700社の要件を束ねた経験
2. **どう作るか設計できる**（要件定義力）→ 300人月のアーキテクチャ刷新
3. **品質の良し悪しが判断できる**（保守運用経験）→ パッケージのライフサイクル管理
4. **AIに的確な指示が出せる**（上流スキル＝プロンプト力）
5. **事業として成立させられる**（PL設計）→ 4.5億の事業を3チームで自走化

**ナレーション**:
「なぜ上流経験者がAI時代に最強なのか。5つあります。

1つ目。何を作るべきかわかる。私はSTRAVISで700社の顧客の要件を束ねてきました。『何が求められているか』を見抜く力は、AIに何を作らせるかを判断する力と同じです。

2つ目。どう作るか設計できる。300人月のアーキテクチャ刷新プロジェクトで、2つのモジュールを統合アプリケーションに再構築しました。この設計力は、AIへの指示の精度に直結します。

3つ目。品質の良し悪しが判断できる。パッケージを保守運用してきた人間は、『これは本番で動かない』が瞬時にわかります。AIが出力したコードの良し悪しを判断できる。

4つ目。AIに的確な指示が出せる。要件定義書を書く力は、実はプロンプトエンジニアリングの本質と同じです。曖昧さを排除し、条件を明確にし、期待するアウトプットを定義する。これを28年やってきました。

5つ目。事業として成立させられる。作るだけでは意味がない。PL設計ができて、チームを組んで、数字で自走させられる。これは純粋なエンジニアにはない力です。」

### スライド6：実演 — プロトタイプ駆動型コンサルティング

**タイトル**: 【実演】業務理解→提案→プロトタイプ→フィードバック を1人で回す
**画面収録**:
（実際のClaude Codeの操作画面を録画 — 4ステップで構成）

**ステップ図（画面に常時表示）**:
```
① 業務を理解する → ② 最適な流れを提案する → ③ プロトタイプを作る → ④ 見せて改善する
    ↑                                                                    ↓
    └──────────────── フィードバックループ ─────────────────────────────────┘
```

**ナレーション**:
「ここで、私が実際にやっている仕事の流れをお見せします。これを『プロトタイプ駆動型コンサルティング』と呼んでいます。

**ステップ1: 業務を理解する。**
一般的なシステム開発者は、要件を『もらう』んです。言われたものを作る。でも私は違います。建設業のバックオフィスがどう動いているか、情報がどこで発生してどこで歪むか、まずそれを自分で理解する。28年の業務コンサルティング経験があるから、これができる。しかも今はAIを使ってリサーチもできるので、一般的なバックオフィスの業務フローを即座に把握して、お客様の業務と比較することもできます。

（Claude Codeでリサーチしている画面）

**ステップ2: 最適なワークフローを提案する。**
お客様の業務を理解した上で、『こういう流れに変えたらどうですか？』と提案する。これが普通のSEにはできないところです。ただ言われた通りに作るのではなく、業務そのものを改善する提案ができる。これがSIPOフレームワークの出番でもあります。

**ステップ3: プロトタイプを作る。**
（Claude Codeでコードを書いている画面）
ここが最大のポイントです。従来のコンサルタントは、提案書……つまり紙で説明していました。でも私は、提案した業務フローを実際に動くプロトタイプとして作ります。お客様に『こんな画面で、こういう操作をすると、こういう結果が出ます』と見せられる。紙の提案書とは説得力がまったく違います。

（出来上がったプロトタイプの画面）

**ステップ4: 見せて改善する。**
動くモノを見せると、お客様から『ここはこうしたい』『この項目も必要だ』と具体的なフィードバックが返ってきます。企画書だけだと抽象的な議論になりがちですが、動くモノがあると議論が具体的になる。そのフィードバックをもとに、また改善する。このループを1人で、しかも数日で回せる。

これが、28年の上流経験とAIが掛け合わさった時に起きることです。当たり前のようにやっていますが、これができる人は実は非常に少ない。業務を理解できて、提案ができて、プロトタイプを自分で作れて、フィードバックを正しく反映できる。この4つが全部できる人は、市場にほとんどいません。」

### スライド7：なぜ「当たり前」が武器になるのか

**タイトル**: あなたの「当たり前」は、実は希少スキル
**比較表**:

| 一般的なシステム開発者 | 上流経験者（プロトタイプ駆動型） |
|---|---|
| 要件を「もらう」 | 業務を理解して「提案する」 |
| 言われた通りに作る | 最適なワークフローを設計する |
| 納品して終わり | フィードバックループを回す |
| 紙の仕様書で合意 | 動くモノで合意 |
| 1つの専門領域 | 企画×設計×開発×業務知識を横断 |

**ナレーション**:
「ここで大事なことをお伝えします。今お見せした4ステップ、私にとっては28年間やってきた当たり前のプロセスです。でも、周りを見渡すと、これが全部できる人は驚くほど少ない。

一般的なシステム開発者は、要件を『もらって』、言われた通りに作って、納品して終わりです。業務を理解して提案することはしない。逆に、戦略コンサルタントは提案はできるけど、自分でプロトタイプは作れない。

上流で企画してきた人間が、AIで自分で作れるようになった。これは、この4ステップを1人で完結できるということです。企画力と業務知識は長年の経験で身についている。作る部分だけがボトルネックだった。そのボトルネックをAIが取り除いた。

だから、あなたが『当たり前にやってきたこと』は、実はAI時代の最強の武器なのです。」

### スライド8：よくある反論への回答

**タイトル**: 「でも、それって…」への回答
**Q&A形式**:

| 反論 | 回答 |
|------|------|
| 「AIが作ったコードは品質が低いのでは？」 | 品質を判断する目を持っているかどうかが重要。上流経験者はそれを持っている |
| 「若い人のほうがAIを使いこなせるのでは？」 | ツールの使い方は若い人が速い。しかし「何を作るか」「業務をどう改善するか」の判断は経験に依存する |
| 「企業で使えるレベルにはならないのでは？」 | プロトタイプの目的は「合意形成」。動くモノを見せて議論を具体化するだけで十分な価値がある |
| 「結局プログラミングは必要では？」 | 必要なのは「読めること」と「判断できること」であって「書けること」ではない |

**ナレーション**:
「よくある反論にお答えします。『AIが作ったコードは品質が低い』。これは半分正解です。だからこそ、品質を判断できる目が重要なのです。パッケージの保守運用を何年もやってきた人間は、ダメなコードが瞬時にわかります。

『若い人のほうが使いこなせる』。ツールの使い方を覚えるスピードは確かに若い人が速い。しかし、『何を作るべきか』『この業務をどう改善すべきか』の判断は、10年、20年の経験に裏打ちされます。お客様の業務を理解して、最適な提案をして、プロトタイプを作って、フィードバックを正しく受け取る。このサイクルには経験が不可欠です。

『企業で使えるレベルにはならない』。プロトタイプの目的は本番リリースではなく、合意形成です。動くモノを見せることで議論が具体的になり、手戻りが激減する。企画書だけで3ヶ月かかっていた合意形成が、プロトタイプがあれば2週間で済む。それだけで十分な価値があります。」

**ナレーション**:
「最後に、キャリア戦略の話をします。従来のコンサルタントは、語って提案して、受注を待ちます。AI活用コンサルタントは、語るだけでなく、プロトタイプを作ってデモで説得できます。そして事業構築型は、語って、作って、自分で売ります。私は今、3番目のフェーズにいます。Kindle 40冊、GitHub 140リポジトリ、STORESでの販売。これが50代のキャリアの新しい形だと思っています。」
