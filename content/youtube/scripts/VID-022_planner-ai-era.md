## この動画で伝えたいこと

28年間「企画して、設計して、人に作ってもらう」側だった人間が、AIの登場で「自分で全部作れる」ようになった。
これは単なるツールの話ではなく、キャリアの構造変化。上流経験者こそAI時代の最適解を出せる。

### スライド4：なぜ上流経験者が最強なのか

**タイトル**: 上流経験者が AI時代に最強な5つの理由
**箇条書き**:
1. **何を作るべきかわかる**（企画力）→ STRAVIS 700社の要件を束ねた経験
2. **どう作るか設計できる**（要件定義力）→ 300人月のアーキテクチャ刷新
3. **品質の良し悪しが判断できる**（保守運用経験）→ パッケージのライフサイクル管理
4. **AIに的確な指示が出せる**（上流スキル＝プロンプト力）
5. **事業として成立させられる**（PL設計）→ 4.5億の事業を3チームで自走化

**ナレーション**:
「なぜ上流経験者がAI時代に最強なのか。5つあります。

1つ目。何を作るべきかわかる。私はSTRAVISで700社の顧客の要件を束ねてきました。『何が求められているか』を見抜く力は、AIに何を作らせるかを判断する力と同じです。

2つ目。どう作るか設計できる。300人月のアーキテクチャ刷新プロジェクトで、2つのモジュールを統合アプリケーションに再構築しました。この設計力は、AIへの指示の精度に直結します。

3つ目。品質の良し悪しが判断できる。パッケージを保守運用してきた人間は、『これは本番で動かない』が瞬時にわかります。AIが出力したコードの良し悪しを判断できる。

4つ目。AIに的確な指示が出せる。要件定義書を書く力は、実はプロンプトエンジニアリングの本質と同じです。曖昧さを排除し、条件を明確にし、期待するアウトプットを定義する。これを28年やってきました。

5つ目。事業として成立させられる。作るだけでは意味がない。PL設計ができて、チームを組んで、数字で自走させられる。これは純粋なエンジニアにはない力です。」

### スライド7：よくある反論への回答

**タイトル**: 「でも、それって…」への回答
**Q&A形式**:

| 反論 | 回答 |
|------|------|
| 「AIが作ったコードは品質が低いのでは？」 | 品質を判断する目を持っているかどうかが重要。上流経験者はそれを持っている |
| 「若い人のほうがAIを使いこなせるのでは？」 | ツールの使い方は若い人が速い。しかし「何を作るか」の判断は経験に依存する |
| 「企業で使えるレベルにはならないのでは？」 | 個人開発とプロトタイプ検証のスピードが変わるだけで十分な価値がある |
| 「結局プログラミングは必要では？」 | 必要なのは「読めること」と「判断できること」であって「書けること」ではない |

**ナレーション**:
「よくある反論にお答えします。『AIが作ったコードは品質が低い』。これは半分正解です。だからこそ、品質を判断できる目が重要なのです。パッケージの保守運用を何年もやってきた人間は、ダメなコードが瞬時にわかります。

『若い人のほうが使いこなせる』。ツールの使い方を覚えるスピードは確かに若い人が速い。しかし、『何を作るべきか』『この品質で大丈夫か』の判断は、10年、20年の経験に裏打ちされます。

『企業で使えるレベルにはならない』。その通りです。でも、プロトタイプを1週間で作れるということは、合意形成のスピードが劇的に変わるということです。企画書だけで説得するのと、動くモノを見せるのとでは、まったく違います。」

### スライド8：キャリア戦略 — 50代からの選択肢

**タイトル**: 50代のキャリア戦略 — 語れるだけでなく作れるコンサルタントへ
**3つの選択肢**:
1. **従来型コンサルタント**: 語る・提案する → 受注を待つ
2. **AI活用コンサルタント**: 語る + 作る → デモで説得する
3. **事業構築型**: 語る + 作る + 売る → 自分で事業を回す ← 今ここ

**ナレーション**:
「最後に、キャリア戦略の話をします。従来のコンサルタントは、語って提案して、受注を待ちます。AI活用コンサルタントは、語るだけでなく、プロトタイプを作ってデモで説得できます。そして事業構築型は、語って、作って、自分で売ります。私は今、3番目のフェーズにいます。Kindle 40冊、GitHub 140リポジトリ、STORESでの販売。これが50代のキャリアの新しい形だと思っています。」
