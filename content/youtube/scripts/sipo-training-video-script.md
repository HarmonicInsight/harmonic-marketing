# YouTube動画スクリプト：建設業バックオフィス改革 ── SIPOフレームワーク解説

対象ツール: INMV（Training Studio）でPPTXから動画化
想定尺: 15〜20分
ターゲット: 建設業・ハウジング業の経営者、管理部門責任者、業務改善コンサルタント

---

## スライド構成

### スライド1：タイトル

**タイトル**: 建設業・ハウジング業のバックオフィスが重い本当の理由
**サブタイトル**: IPOではなくSIPOで再設計せよ
**表示**: HARMONIC insight ロゴ

**ナレーション**:
「建設業・ハウジング業のバックオフィスに関わった方なら、こんな経験はないでしょうか。システムを導入したのにExcel管理が減らない。帳票を整えたのに確認作業が増えた。入力ルールを決めたのに、人によってバラバラ。今日は、この問題の根本原因と、その解決策をお話しします。」

---

### スライド2：よくある症状

**タイトル**: バックオフィスの典型的な症状
**箇条書き**:
- システムを入れたのにExcel管理が減らない
- 帳票を整えたのに確認作業が増えた
- 入力ルールを決めたのに人によってバラバラ
- 部門間で数字が合わない
- 月末の締め処理に追われる

**ナレーション**:
「これらの症状、どれも見覚えがあると思います。多くの会社はこれを『担当者の能力不足』や『システムが悪い』と考えます。しかし、本当の原因は別のところにあります。」

---

### スライド3：IPOの限界

**タイトル**: 従来のIPO設計の限界
**図解**: Input → Process → Output の矢印図
**例**:
- Input：契約情報を入れる
- Process：承認する
- Output：発注書を出す

**ナレーション**:
「従来の業務設計はIPO、つまりインプット・プロセス・アウトプットで行われてきました。入力して、処理して、出力する。一見きれいに見えます。しかし、ここに大きな落とし穴があります。」

---

### スライド4：Inputの正当性問題

**タイトル**: そのInput、本当に正しいですか？
**箇条書き（赤字で強調）**:
- その契約情報は営業が口頭で聞いたものか？
- 見積書の最新版なのか？
- 顧客承認済みなのか？
- 変更合意後なのか？
- 設計図と整合しているのか？

**ナレーション**:
「問題は、Inputの『項目』は定義できても、そのInputが『本当に正しいのか』を保証する仕組みがないことです。営業のファイルと基幹システムで数字が違う。議事録と覚書で条件が違う。これは、情報の発生源、つまりSourceが定義されていないから起きるのです。」

---

### スライド5：SIPOの導入

**タイトル**: SIPOフレームワーク
**図解**: Source → Input → Process → Output の矢印図（Sを大きく強調）

**ナレーション**:
「そこで提案したいのが、SIPOフレームワークです。従来のIPOの前にS、Sourceを加えます。Sourceとは、情報が最初に発生した場所、人、契機、そして原本のことです。単なる元データではありません。『どの事実を正とするか』を決めるものです。」

---

### スライド6：Sourceとは何か

**タイトル**: S：Source ── 情報の発生源
**表形式**:

| Source | 説明 |
|--------|------|
| 顧客要望 | 口頭・メール・打合せメモで発生 |
| 契約書 | 金額・条件の根拠 |
| 見積承認 | 顧客合意金額の原本 |
| 設計図面 | 仕様・数量の根拠 |
| 現場実績 | 進捗・出来高・不具合の事実 |
| 検査結果 | 品質判定の根拠 |
| 引渡確認 | 売上計上・保証開始の起点 |

**ナレーション**:
「建設業・ハウジング業の代表的なSourceを一覧にしました。ポイントは、それぞれの情報について、『何を正とするか』を明確に決めることです。たとえば見積は、営業のファイルが正なのか、基幹システムが正なのか。これが決まっていない会社は、その後の処理が全て不安定になります。」

---

### スライド7：Sourceが抜けると何が起きるか

**タイトル**: Source不在が引き起こす6つの問題
**箇条書き（問題の連鎖を矢印で表現）**:
1. 二重入力
2. 確認作業の増加
3. 部門間の責任転嫁
4. 数字の不一致
5. 手戻り
6. 締め処理の遅延

**中央に大きく**: 「情報源が曖昧な会社ほど属人化する」

**ナレーション**:
「Sourceが定義されていないと、このような問題が連鎖的に起きます。そして最終的に、人が頑張って埋めるしかなくなります。つまり、属人化の本当の原因は、担当者の問題ではなく、情報設計の問題なのです。」

---

### スライド8：理想のバックオフィス4層モデル（概要）

**タイトル**: 理想のバックオフィス4層モデル
**図解（4層を積み重ねた図）**:

```
┌─────────────────────────────────────┐
│  第4層：Output経営活用層             │
│  帳票ではなく意思決定の武器          │
├─────────────────────────────────────┤
│  第3層：Process標準化層              │
│  処理の速さではなく再現性            │
├─────────────────────────────────────┤
│  第2層：Input統制層                  │
│  集めるのではなく受け渡す仕組み      │
├─────────────────────────────────────┤
│  第1層：Source管理層                 │
│  原本を決め、発生責任を定義          │
└─────────────────────────────────────┘
```

**ナレーション**:
「理想のバックオフィスを4層モデルで整理します。大事なのは、一番下のSource管理層です。ここがしっかりしていないと、上の3層は全て崩れます。バックオフィスは事務処理部門ではなく、現場・営業・設計・調達・経理をつなぐ情報統制層であるべきです。」

---

### スライド9：第1層 ── Source管理層

**タイトル**: 第1層：Source管理層
**決めるべき5つのこと**:
1. 何が原本か
2. 誰が発生責任を持つか
3. いつ発生するか
4. どの条件で確定するか
5. 変更時は何をもって更新とするか

**ナレーション**:
「Source管理層で決めるべきことは5つです。何が原本か、誰が発生責任を持つか、いつ発生するか、どの条件で確定するか、そして変更時は何をもって更新とするか。この5つが曖昧な会社は、後工程で必ず混乱します。」

---

### スライド10：第2層 ── Input統制層

**タイトル**: 第2層：Input統制層
**箇条書き**:
- 同じ情報を何度も入力しない
- 入力者を増やしすぎない
- 自由記述を減らす
- マスタと紐づける
- 変更履歴を残す

**キーメッセージ**: 「Inputは"集める作業"ではなく、Sourceを正しく受け渡す仕組み」

**ナレーション**:
「第2層はInput統制層です。ポイントは、Inputを『集める作業』と捉えないことです。InputはSourceの正確な写しでなければなりません。転記を減らし、重複入力を止め、変更履歴を残す。これが統制です。」

---

### スライド11：第3層 ── Process標準化層

**タイトル**: 第3層：Process標準化層
**連動ルールの図解**:
- 契約 → 予算 → 発注 → 実績 → 請求 → 回収
- 設計変更 → 金額変更 → 工程変更 → 再承認
- 完了 → 検査 → 引渡 → 売上 → 保証開始

**キーメッセージ**: 「処理の速さではなく、処理の再現性」

**ナレーション**:
「第3層はProcess標準化層です。案件ごとの勘や経験ではなく、標準ルールで処理することが重要です。契約から回収まで、設計変更から再承認まで、完了から保証開始まで。この連動ルールが標準化されていれば、誰がやっても同じ結果になります。」

---

### スライド12：第4層 ── Output経営活用層

**タイトル**: 第4層：Output経営活用層
**経営者が見たい5つの情報**:
1. どの案件で利益が崩れているか
2. 変更契約の未回収はどこか
3. 発注漏れや原価超過はどこか
4. 引渡遅延の兆候はどこか
5. アフター不具合が多い協力会社はどこか

**キーメッセージ**: 「Outputは帳票ではなく、意思決定の武器」

**ナレーション**:
「最後は第4層、Output経営活用層です。作った帳票を保管するだけでは意味がありません。経営者が本当に知りたいのは、利益が崩れている案件、未回収の変更契約、原価超過の兆候です。Outputは帳票ではなく、意思決定の武器でなければいけません。」

---

### スライド13：ハウジング業の特殊性

**タイトル**: ハウジング業で特に重要な6つのSource
**図解（6つを円形に配置）**:
1. 顧客要望 ── 正式要望と相談レベルの分離
2. 契約・変更契約 ── 最終合意の根拠明確化
3. 設計図・仕様書 ── 最新版管理
4. 発注根拠 ── 図面・数量・予算の紐付け
5. 現場実績 ── 組織学習につなげる記録
6. 引渡・アフター受付 ── 責任分界の起点

**ナレーション**:
「特にハウジング業では、顧客接点が多く、変更も頻繁です。そのため、この6つのSourceを明確に定義することが極めて重要です。特に顧客要望については、『正式な要望』と『相談レベル』を分けないと、営業と設計と工事で認識がずれます。」

---

### スライド14：コンサルタントの5ステップ

**タイトル**: コンサルタントが取り組むべき5つのステップ
**箇条書き（番号付き）**:
1. **Sourceの棚卸し** ── 何がどこで誰によって発生するか
2. **原本の決定** ── 複数あるならどれを正とするか
3. **Input最小化** ── 転記を減らし重複入力を止める
4. **Processの標準化** ── 会社の標準ルールに落とし込む
5. **Outputの経営接続** ── 経営に効く数字を出す業務へ変える

**ナレーション**:
「コンサルタントの役割は、業務フローを書くだけではありません。情報の出生地を見つけ、その情報がどこで歪むかを突き止め、SourceからOutputまで責任の鎖を設計することです。この5つのステップを順に進めることで、バックオフィスの根本的な改革が可能になります。」

---

### スライド15：経営者へのメッセージ

**タイトル**: 経営者の皆様へ
**3つのメッセージ（大きな文字で1つずつ表示）**:

> バックオフィス改革で重要なのは、システム導入ではない。
> 情報の発生源を定義し、"何を正とするか"を統一することだ。

> 業務が回らない原因は、担当者の頑張り不足ではない。
> Sourceが定義されていないまま、InputとProcessだけを増やしてきたことにある。

> 理想のバックオフィスとは、事後処理部門ではない。
> 現場を支え、経営判断を支える情報生産ラインである。

**ナレーション**:
「最後に、経営者の皆様に3つのメッセージをお伝えします。バックオフィス改革の本質は、システム導入ではありません。情報の発生源を定義し、何を正とするかを統一すること。そして、バックオフィスを事後処理部門から、経営判断を支える情報生産ラインに変えること。これが、次世代の建設業に必要な情報統制経営です。」
