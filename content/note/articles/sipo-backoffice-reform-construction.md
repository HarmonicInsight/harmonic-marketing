# 建設業・ハウジング業のバックオフィスが重い本当の理由 ── IPOではなくSIPOで再設計せよ

tags: 建設業DX, バックオフィス改革, SIPO, ハウジング業, 業務設計, コンサルティング

---

## はじめに：なぜバックオフィスは「頑張っても楽にならない」のか

建設業・ハウジング業のバックオフィスに関わった方なら、こんな経験があるはずです。

- システムを入れたのに、Excel管理が減らない
- 帳票を整えたのに、確認作業が増えた
- 入力ルールを決めたのに、人によってバラバラ

原因は、処理能力の不足ではありません。
**情報の発生源（Source）が定義されないまま、入力と処理だけを増やしてきたこと**にあります。

この記事では、建設業・ハウジング業のバックオフィスを根本から見直すための考え方として、**SIPO（Source → Input → Process → Output）**フレームワークを紹介します。

---

## 1. IPOで設計してもうまくいかない理由

従来の業務設計はIPO（Input → Process → Output）で行われてきました。

| IPO | 例 |
|-----|-----|
| Input | 契約情報を入れる |
| Process | 承認する |
| Output | 発注書を出す |

一見きれいです。しかし、実際の現場ではこう問題が起きます。

- その契約情報は、**営業が口頭で聞いたもの**なのか
- **見積書の最新版**なのか
- **顧客承認済み**なのか
- **変更合意後**なのか
- **設計図と整合**しているのか

**Inputは定義できても、Inputの正当性を保証するSourceが定義されていない。**

これがIPOの限界です。

---

## 2. SIPOとは何か

SIPOは、IPOの前に**S（Source）**を加えたフレームワークです。

### S：Source（情報の発生源）

情報が**最初に発生した場所・人・契機・原本**。

建設業・ハウジング業での代表的なSourceは以下です。

| Source | 説明 |
|--------|------|
| 顧客要望 | 口頭・メール・打合せメモで発生 |
| 契約書 | 金額・条件の根拠 |
| 見積承認 | 顧客が合意した金額の原本 |
| 設計図面 | 仕様・数量の根拠 |
| 現場実績 | 進捗・出来高・不具合の事実 |
| 検査結果 | 品質判定の根拠 |
| 協力会社報告 | 外注側の実績 |
| 引渡確認 | 売上計上・保証開始の起点 |
| アフター受付 | 不具合・保証対応の起点 |

**Sourceは単なる「元データ」ではありません。「どの事実を正とするか」を決めるもの**です。

### I：Input（取り込み）

Sourceから業務システムや管理帳票に取り込む項目。契約金額、変更金額、着工予定日、発注数量、完了日、検査結果、引渡日、不具合内容など。

**Inputは、Sourceの写しであって、現場や事務が自由に作るものではありません。**

### P：Process（処理）

判断、承認、変換、照合、集計、進捗化、会計処理。契約承認、予算化、発注化、出来高計上、売上計上、原価配賦、変更履歴管理、保証対応判定など。

### O：Output（成果物）

意思決定や現場実行のために使う成果物。工事台帳、発注書、支払一覧、原価予実表、進捗レポート、顧客向け案内、引渡資料、アフター対応一覧、経営ダッシュボードなど。

## 3. なぜSourceが抜けると全てが壊れるのか

建設業やハウジング業で起きている問題の多くは、**Source不在のままInputだけが回っていること**です。

具体的に見てみましょう。

| 問い | 実際に起きること |
|------|----------------|
| 見積は営業ファイルが正か、基幹システムが正か？ | 数字が部門間でずれる |
| 契約後変更は議事録が正か、覚書が正か？ | 変更漏れ・請求漏れが発生 |
| 着工判断は現場所長の連絡が正か、社内承認が正か？ | 手戻り・責任の所在が不明 |
| 出来高は写真か、検査記録か、日報か？ | 計上タイミングがずれる |
| 売上計上は引渡書か、施主検収か？ | 会計処理が遅延する |

Sourceが曖昧な結果、以下が起きます。

- **二重入力**
- **確認作業の増加**
- **部門間の責任転嫁**
- **数字の不一致**
- **手戻り**
- **締め処理の遅延**

**情報源が曖昧な会社ほど、人が頑張って埋めるしかなくなり、属人化します。**

## 4. 理想のバックオフィス4層モデル

理想のバックオフィスは「事務処理部門」ではありません。
**現場・営業・設計・調達・経理をつなぐ"情報統制層"**です。

### 第1層：Source管理層

情報の発生源を定義し、原本を決める層。

決めるべきことは5つです。

1. **何が原本か**
2. **誰が発生責任を持つか**
3. **いつ発生するか**
4. **どの条件で確定するか**
5. **変更時は何をもって更新とするか**

ここがない会社は、全部が後工程で崩れます。

### 第2層：Input統制層

Sourceを、最小限の入力で業務に取り込む層。

- 同じ情報を何度も入力しない
- 入力者を増やしすぎない
- 自由記述を減らす
- マスタと紐づける
- 変更履歴を残す

**Inputは"集める作業"ではなく、Sourceを正しく受け渡す仕組み**です。

### 第3層：Process標準化層

承認、照合、発注、計上、支払、引渡を、案件ごとの勘ではなく、**標準ルールで処理する層**。

必要な連動ルールの例：

- 契約 → 予算 → 発注 → 実績 → 請求 → 回収
- 設計変更 → 金額変更 → 工程変更 → 再承認
- 完了 → 検査 → 引渡 → 売上 → 保証開始

**バックオフィスの理想は、処理の速さではなく、処理の再現性です。**

### 第4層：Output経営活用層

帳票を保管するだけでなく、**経営判断に使える形で出す層**。

経営者が本当に見たい情報：

- どの案件で利益が崩れているか
- 変更契約の未回収はどこか
- 発注漏れや原価超過はどこか
- 引渡遅延の兆候はどこか
- アフター不具合が多い協力会社はどこか

**Outputは帳票ではなく、意思決定の武器でなければいけません。**

## 5. ハウジング業で特に重要な6つのSource

ハウジング業は一般建設よりも顧客接点と変更が多いため、Source定義がさらに重要です。

### (1) 顧客要望
口頭・メール・打合せメモ・営業メモが混在しやすい。
「正式要望」と「相談レベル」を分けることが必要です。

### (2) 契約・変更契約
最終合意の根拠を明確にしないと、営業・設計・工事・経理で数字がずれます。

### (3) 設計図・仕様書
最新版管理が崩れると、現場も発注も請求も崩れます。

### (4) 発注根拠
誰が、どの図面・どの数量・どの予算で発注したか。ここが弱いと原価管理が機能しません。

### (5) 現場実績
進捗、出来高、手戻り、不具合、検査結果。現場個人のメモで終わると組織学習が起きません。

### (6) 引渡・アフター受付
売上計上や保証開始、不具合責任分界の起点になります。

## 6. コンサルタントは何を定義すべきか

コンサルタントの役割は、業務フローを書くだけではありません。
本質的には、**情報の出生地を見つけ、SourceからOutputまで責任の鎖を設計すること**です。

### コンサルタントが取り組むべき5つのステップ

| ステップ | 内容 |
|----------|------|
| 1. Sourceの棚卸し | 何の情報が、どこで、誰によって、どんな形で発生するかを洗い出す |
| 2. 原本の決定 | 同じ情報が複数あるなら、どれを正とするか決める |
| 3. Input最小化 | 現場や事務の転記を減らし、重複入力を止める |
| 4. Processの標準化 | 案件ごとの差ではなく、会社の標準ルールに落とし込む |
| 5. Outputの経営接続 | 帳票を作るための業務から、経営に効く数字を出す業務へ変える |

---

## 7. 経営者へのメッセージ

最後に、建設業・ハウジング業の経営者に伝えたいことを3つにまとめます。

> **バックオフィス改革で重要なのは、システム導入ではない。**
> 情報の発生源を定義し、現場・営業・設計・経理の間で"何を正とするか"を統一することだ。

> **業務が回らない原因は、担当者の頑張り不足ではない。**
> Sourceが定義されていないまま、InputとProcessだけを増やしてきたことにある。

> **理想のバックオフィスとは、事後処理部門ではない。**
> 現場を支え、経営判断を支える情報生産ラインである。

---

## まとめ

建設業・ハウジング業のバックオフィスは、単なる事務処理の効率化では変わりません。

必要なのは、**S（Source）を起点にした情報流通の再設計**です。

SIPOフレームワークと4層モデルを使って、情報の発生源から経営判断までを一本の線でつなぐ。
それが、次世代の建設業に必要な**情報統制経営**です。
