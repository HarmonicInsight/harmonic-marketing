# 建設業に「アジャイル」は本当に必要か｜RPAもDXも同じ轍を踏んでいる

## また同じ話が始まった

「建築業界でアジャイル開発の認知度を調査しました。知っている人は50%でした。」

2025年3月、あるIT企業がこんな調査結果を発表しました。

私はこれを見て、正直こう思いました。

**「また始まったか。」**

RPA、DX、AI、そしてアジャイル。IT業界で流行ったバズワードが、数年遅れで建設業に持ち込まれるパターンを、もう何度も見てきました。そして毎回、同じことが起きます。

1. 「○○を知っていますか？」という認知度調査が出る
2. 「知らない人が多い。導入が遅れている」と危機感を煽る
3. 「導入しましょう」とツールやコンサルが売り込まれる
4. 現場が混乱して、数年後に「失敗した」と言われる

この記事では、なぜこのパターンが繰り返されるのか、そして建設業が本当にやるべきことは何かを書きます。

---

## 「知っているか」を聞いても意味がない

まず、調査そのものの問題です。

「アジャイル開発を知っていますか？」——これに「知っている」と答えた50%の人は、何を知っているのでしょうか。

- 「スプリントという2週間の繰り返しでやる手法ですよね」
- 「ウォーターフォールの反対のやつですよね」
- 「朝会やるやつですよね」

おそらくこの程度の理解でしょう。それは「知っている」とは言いません。

アジャイルの本質は、手法ではなく**思想**です。

- 計画通りに進めることより、**変化に対応すること**を重視する
- ドキュメントより、**動くもの**を重視する
- 契約交渉より、**顧客との協働**を重視する
- 完璧な計画を立てるより、**小さく試してフィードバックを得ること**を重視する

この思想を理解し、実践できている人がどれだけいるか。**IT業界ですら、ほとんどいません。**

---

## IT業界でもアジャイルは「できていない」

私はIT業界で長くコンサルティングをしてきました。断言します。

**アジャイルをまともに実践できているチームは、IT業界でも数%以下です。**

よくある「自称アジャイル」の実態：

| 表面上やっていること | 実態 |
|---|---|
| 2週間スプリント | ただの2週間区切りのウォーターフォール |
| デイリースタンドアップ | 朝会という名の進捗報告会 |
| スクラムマスター配置 | 肩書きだけの進捗管理者 |
| バックログ管理 | Excelの課題一覧と何も変わらない |
| レトロスペクティブ | 「次は頑張ります」で終わる反省会 |

なぜこうなるか。アジャイルが前提としている条件が揃っていないからです。

### アジャイルが機能するための前提条件

1. **プロダクトオーナーが意思決定権を持っている**（日本企業では稟議・合議が多く、即断即決できない）
2. **チームが自律的に動ける**（上司の承認なしに技術的判断ができる）
3. **手戻りのコストが小さい**（ソフトウェアは書き直せるが、建物は壊せない）
4. **顧客が協働してくれる**（2週間ごとにフィードバックをくれる顧客がどれだけいるか）
5. **「完成」の定義を柔軟に変えられる**（法規制や安全基準がある業界では不可能）

この5つのうち、日本の一般的な企業で揃っているのは、よくて1つか2つ。IT業界ですらそうなのに、建設業で揃うはずがありません。

## 建設業にアジャイルを持ち込む矛盾

建設業の特性を考えてみましょう。

**「小さく作って試す」ができない。**

ソフトウェアなら、プロトタイプを作ってユーザーに見せて、「違う」と言われたら書き直せます。コストは開発者の時間だけです。

建設業で「小さく建てて、住んでみて、違ったら壊してやり直す」は物理的に不可能です。図面の変更1つで数千万円のコストが動く世界で、「変化を歓迎しましょう」とは言えません。

**法規制で「柔軟な変更」ができない。**

建築基準法、消防法、労働安全衛生法。建設業は法規制の塊です。「スプリントレビューで方針が変わったので構造を変えます」は許されません。

**責任の所在が曖昧になる。**

アジャイルは「チームで責任を持つ」思想ですが、建設業では施工管理者、設計者、元請け、下請けの責任が明確に分かれています。これは法的にそうなっている。「みんなで決めました」では済まない世界です。

## ではRPAの時は何が起きたか

同じことがRPAの時にも起きました。

2018年頃、「RPAで建設業のバックオフィスを自動化しましょう」というブームが来ました。

結果どうなったか。

- RPAツールを買ったが、自動化する業務を選定できなかった
- 月に数回しかやらない業務を自動化して、ROIが出なかった
- ロボットを作ったが、業務システムが更新されて動かなくなった
- 「RPAは使えない」という結論になった

**RPAが使えなかったのではなく、RPAを入れる前の業務整理をしなかったから失敗した**のです。

DXも同じです。「DXしましょう」と言われてシステムを導入したが、業務プロセスが旧来のままなので、システムだけが浮いている。

そして今、アジャイル。

## バズワードの前にやるべきこと

RPA、DX、AI、アジャイル。手法の名前は変わっても、建設業が本当にやるべきことは、ずっと同じです。

### やるべきこと: 業務を構造的に理解する

「アジャイルを知っていますか？」ではなく、こう聞くべきです。

**「あなたの業務を、インプットとアウトプットで説明できますか？」**

- この業務のトリガー（起点）は何か
- 何が入力され、何が出力されるか
- 判断が必要なポイントはどこか
- 例外が発生するのはどんな時か
- その業務は月に何回発生するか

これが説明できないなら、どんな手法を導入しても機能しません。アジャイルだろうがウォーターフォールだろうが、**業務の解像度が低い状態で手法を選んでも意味がない**のです。

### SIPOで考える

私はSource-Input-Process-Outputの4層（SIPO）で業務を分解することを提唱しています。

```
Source （情報源）  : どこからデータ/指示が来るか
  ↓
Input  （入力）    : 何を受け取って作業が始まるか
  ↓
Process（処理）    : 何をどう判断・加工するか
  ↓
Output （出力）    : 何を誰に渡すか
```

この4層で整理すると、自動化すべき部分が見えてきます。

- Sourceが不安定（口頭指示、電話メモ）→ まずSourceを安定させる
- Processに人間の判断が多い → すぐには自動化できない
- Outputが定型 → 自動化の効果が出やすい

アジャイルもRPAもAIも、この整理の**後に**検討すべきものです。順番が逆になっている限り、何を入れても失敗します。

## 建設業が学ぶべきは「手法」ではなく「段階的改善」の考え方

とはいえ、アジャイルの思想から学べることはあります。ただし、それは「スプリント」や「スクラム」といった手法ではありません。

学ぶべきは**「段階的に改善する」という考え方そのもの**です。

- 全業務を一気に変えようとしない
- 1つの業務で効果を確認してから横展開する
- 現場のフィードバックを取り入れながら進める
- 完璧な計画を立ててから始めるのではなく、小さく始める

これは別にアジャイルと呼ばなくても、**PDCA**とも**カイゼン**とも言えます。建設業が昔からやってきたことです。

新しいラベルを貼る必要はありません。やるべきことは昔から変わっていない。

---

## まとめ: 手法を導入する前に、業務を説明できるようになれ

建設業に限らず、すべての業界に共通することです。

- 「アジャイルを知っていますか？」→ **聞く意味がない**
- 「アジャイルを導入しましょう」→ **前提条件が揃っていなければ失敗する**
- 「業務を構造的に説明できますか？」→ **これが出発点**

RPA、DX、AI、アジャイル。流行る手法は変わっても、**業務を理解しないまま手法を入れれば失敗する**という原則は、10年前も今も10年後も変わりません。

バズワードに振り回される前に、まず自分の業務を4層で書き出してみてください。それだけで、何をすべきかが見えてきます。
