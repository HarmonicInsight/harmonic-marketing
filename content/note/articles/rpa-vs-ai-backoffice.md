# RPAとAIの本質的な違い ── バックオフィスで「AIは使えない」と言われる本当の理由

tags: RPA, AI, バックオフィス改革, IPO分析, SIPO, ハウジング業, 業務改善, コンサルティング

---

## はじめに：ハウジング会社のバックオフィスで見た現実

ハウジング会社のバックオフィス業務コンサルティングをしていると、ある共通の反応に出会います。

**「AIって結局、うちでは使えないですよね」**

この言葉の裏には、単なる拒否反応ではない、構造的な理由があります。

彼らはExcelや基幹システムを、決められた作業手順に従ってこなしていく。契約情報を入力し、承認を通し、発注書を出し、原価を集計する。この一連の手順を正確にたどることが、バックオフィスの価値そのものです。

RPAはこの手順をそのまま自動化できたから、バックオフィスと相性が良かった。

一方AIは、その手順をすっ飛ばして、いきなり結論を出そうとする。

途中の作業手順が見えない。検証できない。だから「使えない」。

**これは現場の無理解ではありません。AIの適用方法が間違っているのです。**

---

## 1. RPAとAIは何が違うのか

RPAとAIを並べて議論する場面は多いですが、この2つは自動化する「対象」が根本的に異なります。

| | RPA | AI |
|---|---|---|
| **自動化の対象** | 作業手順（操作の連続） | 判断・生成（思考の代替） |
| **前提** | 手順が明確に定義されている | 手順は不要、入出力だけあればよい |
| **得意な場面** | 決まった画面を決まった順番で操作する | 非定型な情報から結論を出す |
| **現場の安心感** | 高い（途中経過が見える） | 低い（途中が見えない） |
| **失敗時の対応** | どの手順で止まったかわかる | なぜその結論になったか説明が難しい |

### RPAが好まれた理由

バックオフィス担当者にとって、RPAは「自分の仕事をそのままロボットがやってくれる」存在でした。

- システムAにログインする → ○
- 所定のメニューを開く → ○
- Excelからデータをコピーする → ○
- 入力欄に貼り付ける → ○
- 確認ボタンを押す → ○

**手順が100%可視化されている。** だから信頼できた。途中で止まっても「ステップ7で失敗した」とわかるから、対処もできた。

### AIが嫌われる理由

一方、AIに同じ業務を任せようとすると、こうなります。

- 「契約情報を渡したら、発注書を自動生成してくれる」

一見便利です。しかし現場はこう感じます。

- 承認プロセスは通ったのか？
- 予算との照合はしたのか？
- 変更契約の反映はされているのか？
- どの見積を根拠にしたのか？

**AIは結論を出すが、そこに至る手順を見せない。**

バックオフィスにとって手順は「省略していいもの」ではなく、**業務の正当性を担保するもの**です。手順が見えないことは、単に不便なだけでなく、**業務として成立しない**ことを意味します。

## 2. なぜ「AIは使えない」が正しく、かつ間違いなのか

### 正しい面：AIを業務全体に適用するのは間違い

バックオフィスの業務フロー全体をAIに置き換えようとするのは、確かに間違いです。

業務には手順遵守が必要な工程があり、それを飛ばすことは許されません。承認なしの発注、照合なしの支払い、確認なしの計上──これらはAIの判断だけでは成立しません。

### 間違いの面：AIが使えるポイントは必ず存在する

しかし、業務フローの中には「判断」や「生成」を求められる工程が必ずあります。

- 契約書を読んで要点を抽出する
- 過去の類似案件を検索して見積根拠を見つける
- 議事録を要約する
- 帳票間の数値の整合性をチェックする
- レポートの下書きを作成する

**これらの工程にピンポイントでAIを適用すれば、手順は守りつつ、処理速度と品質が上がります。**

問題は「AIを使うか使わないか」ではなく、**「業務のどこにAIを使うか」を分析していないこと**にあります。

## 3. IPO分析こそがAI活用の前提条件

ここで登場するのが、**IPO分析**です。

### IPO（Input → Process → Output）とは

業務を3つの要素に分解するフレームワークです。

| 要素 | 意味 | 例 |
|------|------|-----|
| **I（Input）** | 業務に投入される情報・データ | 契約書、見積書、Excel台帳 |
| **P（Process）** | 入力に対して行われる処理 | 承認、照合、計算、変換、判断 |
| **O（Output）** | 処理の結果として出される成果物 | 発注書、レポート、集計表 |

### AI時代のIPO分析の核心

**I（入力）とO（出力）は人間が定義する。P（処理）の中でAIが使えるかを判断する。**

これがAI時代の業務改善の正しいアプローチです。

具体的に見てみましょう。

#### 例1：見積査定業務

| 要素 | 内容 | AIの適用 |
|------|------|---------|
| I | 協力会社からの見積書PDF | - |
| P① | PDFを読み取り、項目・単価・数量を抽出する | **→ AI適用可** |
| P② | 過去の同類案件の単価と比較する | **→ AI適用可** |
| P③ | 差異が大きい項目にフラグを立てる | **→ AI適用可** |
| P④ | 上長が承認判断を行う | → 人間が行う |
| O | 査定結果レポート + 承認済み見積 | - |

#### 例2：月次原価集計業務

| 要素 | 内容 | AIの適用 |
|------|------|---------|
| I | 各案件の発注実績・出来高・支払データ | - |
| P① | データを案件別に集計する | → RPA or マクロ |
| P② | 予算との乖離を分析し、コメントを生成する | **→ AI適用可** |
| P③ | 異常値の原因を過去データから推測する | **→ AI適用可** |
| P④ | 経理担当が確認・承認する | → 人間が行う |
| O | 月次原価レポート | - |

**ポイントは、Pを細分化し、その中でAIが担えるサブプロセスを特定すること**です。

## 4. RPAとAIの正しい使い分け

IPO分析を行うと、RPAとAIの使い分けが明確になります。

### P（Process）の分類と適用技術

| Pの種類 | 特徴 | 適用技術 |
|---------|------|---------|
| **手順型P** | 決まった操作を決まった順番で行う | **RPA** |
| **判断型P** | 情報を読み取り、判断・分類・評価する | **AI** |
| **生成型P** | 情報から文書・レポート・要約を作成する | **AI** |
| **照合型P** | 複数データを突き合わせて一致/不一致を判定する | **AI or RPA**（複雑さによる） |
| **承認型P** | 責任を伴う意思決定 | **人間** |

### RPA + AIの協調モデル

最も効果的なのは、RPAとAIを組み合わせるモデルです。

```
[手順型Pの自動化] ── RPA
        ↓
[判断型・生成型Pの自動化] ── AI
        ↓
[承認型P] ── 人間
```

RPAがデータを集め、AIが分析・判断し、人間が承認する。

**この三層構造が、AI時代のバックオフィスのあるべき姿です。**

## 5. SIPOで見ればさらに精密になる

IPO分析をさらに進化させたのが、**SIPO（Source → Input → Process → Output）**フレームワークです。

IPOの前にS（Source＝情報の発生源）を加えることで、「そもそもInputの根拠は何か」まで遡って設計できます。

### RPA × AI × SIPO の統合モデル

| 層 | 役割 | 技術 |
|----|------|------|
| **S（Source）** | 情報の発生源を定義・管理 | マスタ管理・ルール定義（人間） |
| **I（Input）** | Sourceから業務へ取り込み | RPA / API連携 |
| **P（Process）** | 処理・判断・生成 | RPA（手順型） + AI（判断型・生成型） |
| **O（Output）** | 成果物の出力・配布 | RPA / 自動配信 |

**Sourceが定義されていれば、Inputの自動化（RPA）も、Processの知能化（AI）も、初めて安全に実装できます。**

逆に言えば、Sourceが曖昧なままAIを入れても、「何を根拠にその判断をしたのか」が検証できず、結局「使えない」に戻ります。

## 6. 現場への導入ステップ

### ステップ1：業務棚卸し（IPO分析）

まず、対象業務をIPOで分解します。

- I：何が入力されるか
- P：何が行われるか（手順を細分化）
- O：何が出力されるか

### ステップ2：Pの分類

Pを「手順型」「判断型」「生成型」「照合型」「承認型」に分類します。

### ステップ3：技術マッピング

- 手順型P → RPA候補
- 判断型・生成型P → AI候補
- 承認型P → 人間が継続

### ステップ4：Source確認（SIPO拡張）

各Inputの根拠（Source）が明確かを確認します。Sourceが曖昧な業務は、自動化の前にSource定義から始めます。

### ステップ5：小さく始める

最も効果が高く、リスクが低いPから始めます。いきなり業務全体をAI化しようとしない。

## 7. 経営者・管理者へのメッセージ

> **「AIは使えない」のではなく、「AIの使い方を間違えている」のです。**

> **業務全体をAIに丸投げするのではなく、IPO分析でProcessを分解し、AIが効くポイントだけにピンポイントで適用する。**

> **RPAとAIは競合ではなく補完関係にある。手順の自動化はRPA、判断の自動化はAI、責任の意思決定は人間。この三層構造がAI時代のバックオフィスの正解です。**

## まとめ

| テーマ | 結論 |
|--------|------|
| RPAとAIの違い | 手順の自動化 vs 判断の自動化 |
| AIが嫌われる理由 | 手順を飛ばすから（検証できない） |
| 正しいAI活用 | IPO分析でPを分解し、判断型Pに適用 |
| 最適モデル | RPA（手順型P）+ AI（判断型P）+ 人間（承認型P） |
| さらに精密に | SIPOでSourceから設計する |

**AI時代の業務改善とは、AIを導入することではなく、業務を正しく分析した上で、AIが効く場所を見つけることです。**
