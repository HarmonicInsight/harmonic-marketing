## はじめに

「消し込みに毎月3日かかっている」
「月末になると経理が帰れない」
「あの入金、どの工事の分？と毎回聞かれる」

建設業の経理担当から、こういった声を何度も聞いてきました。

消し込み——つまり請求と入金の照合、あるいは債務と支払の突合——は、どの会社でも毎月必ず発生する業務です。しかし建設業では、他業種では考えられないほど複雑になります。

なぜか。そして、DXで本当に解決できるのか。

結論を先に言います。

> **消し込みの問題は、消し込みシステムでは解決しない。**

---

## 建設業の消し込みが複雑になる理由

一般的な物販業であれば、消し込みはシンプルです。

```
請求: 100万円 → 入金: 100万円 → 消込完了
```

しかし建設業では、こうなります。

```
出来高請求: 1,000万円
入金: 800万円（査定減額200万）
しかも安全協会費15万と振込手数料880円が差し引かれている
さらに前払金500万の充当分がある
```

1つの工事でこれですから、月に数十件の工事が動いていれば、月末に何が起きるかは想像がつくでしょう。

---

## どの会社も同じ5つの壁にぶつかる

業種も規模も違う建設会社を見てきましたが、消し込みで詰まるポイントは驚くほど共通しています。

### 壁①：工事番号がつながらない

受注管理では「A-001」、原価管理では「2026-東京-01」、経理では「請求No.5678」。

同じ工事なのに、システムや部署ごとに別の番号を使っている。これだと、入金があっても「これはどの工事の分？」がすぐにわかりません。

**消し込みの自動化以前に、番号がつながっていなければ照合のしようがない**のです。

### 壁②：伝票が主語になっている

本来、業務は「契約した → 出来高が進んだ → 請求した → 入金された」という**イベントの流れ**です。

しかし実務では「伝票を入力する」が仕事の単位になりがちです。その結果、「なぜこの金額なのか」「どの出来高に対応するのか」という文脈が伝票に残りません。

消し込みの時に困るのは、まさにこの文脈の欠落です。

### 壁③：例外が多すぎる

建設業では以下が「日常」です。

- 一部支払（出来高の一部だけ入金）
- 相殺（安全協会費、保証金等を差し引いて入金）
- 前払充当（着工時前払金を完成時に充当）
- 保留（検査未了、瑕疵対応等で一部を保留）

これらは「例外」ではなく、建設業の取引構造上の**標準的なパターン**です。しかし多くのシステムは、この「標準的な例外」に対応しきれていません。

### 壁④：月末に集中する

消し込みは「できたときにやる」のが理想ですが、実際には月末の銀行入金データが届いてから一気にやる会社がほとんどです。

そのため、月末〜翌月5日に業務が集中し、特定の担当者しか処理できない**属人化の温床**になっています。

### 壁⑤：現場と経理の「見たいもの」が違う

現場所長は「工区ごとの原価」「業者ごとの支払状況」が見たい。
経理は「勘定科目ごとの月次残高」が見たい。

この粒度の違いを放置したままシステムを入れると、**現場が使わない会計システム**か**会計につながらない現場管理**のどちらかが生まれます。

---

## DXの前に整えるべき3つのこと

では、何をすればいいのか。答えはシンプルです。

### 1. 工事番号体系を統一する

受注・原価・請求・支払・消し込みで、**同じ工事を同じ番号で追えるようにする**。

これが整っていれば、たとえExcel管理でも消し込みの精度は劇的に上がります。逆にこれが整っていなければ、どんな高機能なシステムを入れても回りません。

### 2. 業務の流れを2本定義する

建設業の取引は、突き詰めると2本の流れに整理できます。

**売上側**: 受注 → 出来高 → 請求 → 入金 → 入金消込
**仕入側**: 発注 → 検収 → 支払請求受領 → 支払 → 支払消込

この2本の流れに沿って「いつ・誰が・何のデータを作るか」を定義すれば、消し込み時点で必要な情報が揃う設計になります。

### 3. 例外パターンをルール化する

一部支払、相殺、前払充当、保留——これらを「例外」として放置するのではなく、**パターンとして分類し、それぞれの処理ルールを事前に決める**。

例えば:
- 振込手数料880円以内の差額は自動調整
- 安全協会費は指定の勘定科目に自動振替
- 一部入金は古い請求から順に充当（FIFO）

ルールが明文化されていれば、自動化も属人化解消も実現可能です。

## 正しい順序で進める

多くのDXプロジェクトが失敗する原因は、**順序の間違い**です。

```
❌ よくある失敗パターン:
   AI-OCR導入 → ワークフロー構築 → 番号体系の整理（後回し）

✅ 正しい順序:
   番号体系の統一 → 業務フロー定義 → ルール策定 → システム化
```

AI-OCRで請求書を読み取れても、紐づけ先の工事番号が統一されていなければ意味がありません。ワークフローで承認を効率化しても、元データの粒度が揃っていなければ経理が苦しむだけです。

**DXは「速くする」技術ではなく、「正しい構造を作る」行為**です。

---

## まとめ

建設業の消し込みが回らないのは、消し込みの問題ではありません。

伝票処理・受注処理・支払処理・消し込みが、**同じ工事の流れとして設計されていないこと**が本当の原因です。

そして、ボトルネックは明確です。

> **工事単位で受注から支払・消し込みまでを一気通貫で紐づける共通キーと、例外を吸収できる業務ルールがないこと。**

DXの第一歩は、システム選定ではありません。
まず、自社の工事番号体系を棚卸しすることから始めてください。
